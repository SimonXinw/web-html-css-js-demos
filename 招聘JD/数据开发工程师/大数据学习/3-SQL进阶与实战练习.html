一、SQL 执行顺序（必须透彻理解） 面试必考
SQL 语句的真实执行顺序 必背
我们书写 SQL 的顺序是 SELECT → FROM → WHERE → ...，但数据库引擎内部的执行顺序完全不同。理解执行顺序是写出正确 SQL、理解别名作用域的基础。
① FROM → ② WHERE → ③ GROUP BY → ④ HAVING → ⑤ SELECT → ⑥ ORDER BY → ⑦ LIMIT
  • FROM：确定数据来源，执行 JOIN、笛卡尔积，生成虚拟表
  • WHERE：对虚拟表逐行过滤，不满足条件的行被丢弃
  • GROUP BY：将剩余行按指定列分组，每组生成一行
  • HAVING：对分组后的结果过滤（可使用聚合函数）
  • SELECT：计算需要输出的列、表达式、别名
  • ORDER BY：对最终结果排序（可使用 SELECT 中的别名）
  • LIMIT：截取指定行数返回
为什么 WHERE 中不能用别名？
WHERE 在 SELECT 之前执行，此时别名尚未定义，引擎不认识该名称。
错误写法 SELECT amount * 0.9 AS discounted FROM orders WHERE discounted > 100 -- 报错！discounted 尚未定义
正确写法 SELECT amount * 0.9 AS discounted FROM orders WHERE amount * 0.9 > 100 -- 直接使用原始表达式
为什么 HAVING 可以用聚合函数？
HAVING 在 GROUP BY 之后执行，此时分组已完成，聚合函数有了计算上下文。
示例：找出月订单数超过 10 的用户 SELECT user_id, COUNT(*) AS order_cnt FROM orders WHERE dt >= "2024-01-01" GROUP BY user_id HAVING COUNT(*) > 10 -- HAVING 可以用聚合函数 ORDER BY order_cnt DESC -- ORDER BY 可以用 SELECT 别名

二、窗口函数进阶 数据分析核心武器
NTILE(n)：等分桶 核心
将有序数据等分为 n 组，返回每行所在的组号（1 ~ n）。常用于分位数计算、用户分层。
示例：将用户按消费金额分为 4 个等级 SELECT user_id, total_amount, NTILE(4) OVER (ORDER BY total_amount DESC) AS tier FROM user_stats -- tier=1 是 Top 25% 高消费用户 -- tier=4 是 Bottom 25% 低消费用户
FIRST_VALUE / LAST_VALUE
取窗口内第一个或最后一个值。LAST_VALUE 需注意默认窗口范围只到当前行。
示例：每个类目中价格最高和最低的商品名 SELECT category, product_name, price, FIRST_VALUE(product_name) OVER ( PARTITION BY category ORDER BY price DESC ) AS most_expensive, FIRST_VALUE(product_name) OVER ( PARTITION BY category ORDER BY price ASC ) AS cheapest FROM products
ROWS BETWEEN 窗口范围详解 必背
窗口范围决定了聚合函数计算时包含哪些行，是滑动窗口计算的核心。
  • UNBOUNDED PRECEDING：窗口起点 = 分区第一行
  • CURRENT ROW：当前行
  • n PRECEDING：当前行往前 n 行
  • n FOLLOWING：当前行往后 n 行
  • UNBOUNDED FOLLOWING：窗口终点 = 分区最后一行
常见窗口范围组合 -- 从分区起始到当前行（累计求和默认范围） ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW -- 前 6 行 + 当前行 = 最近 7 天（需按日期排序） ROWS BETWEEN 6 PRECEDING AND CURRENT ROW -- 整个分区（求全局占比时使用） ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
滑动平均实战：最近 7 天平均 GMV
电商日报中常见需求：计算每天的 GMV 以及最近 7 天滑动平均 GMV，消除日波动。
完整 SQL SELECT dt, daily_gmv, AVG(daily_gmv) OVER ( ORDER BY dt ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS gmv_7d_avg, SUM(daily_gmv) OVER ( ORDER BY dt ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS gmv_7d_sum FROM ( SELECT dt, SUM(pay_amount) AS daily_gmv FROM dwd_order WHERE status = "paid" GROUP BY dt ) t ORDER BY dt
累计百分比：帕累托分析（二八法则）
计算每个商品的销售额及其占总销售额的累计百分比，用于判断哪些商品贡献了 80% 的销售额。
帕累托分析 SQL SELECT product_id, product_name, sales_amount, SUM(sales_amount) OVER (ORDER BY sales_amount DESC) AS cumulative_sales, ROUND( SUM(sales_amount) OVER (ORDER BY sales_amount DESC) / SUM(sales_amount) OVER () * 100, 2 ) AS cumulative_pct FROM ( SELECT product_id, product_name, SUM(pay_amount) AS sales_amount FROM dwd_order_detail GROUP BY product_id, product_name ) t ORDER BY sales_amount DESC -- cumulative_pct 达到 80% 时对应的商品就是核心商品
窗口函数速查表
函数作用典型场景
ROW_NUMBER()不重复序号 1,2,3分组取 TopN
RANK()并列跳号 1,1,3排名有并列需求
DENSE_RANK()并列不跳号 1,1,2密集排名
NTILE(n)等分 n 桶分位数 / 分层
LAG(col, n)往前取 n 行环比计算
LEAD(col, n)往后取 n 行下一次行为间隔
FIRST_VALUE窗口第一个值分组最值
SUM() OVER窗口累计求和累计 GMV / 累计百分比
AVG() OVER窗口滑动平均7 日 / 30 日滑动平均

三、复杂 JOIN 实战 真实场景中的多表关联
自关联（Self Join） 核心
找出同一天下过两笔以上订单的用户。自关联即一张表 JOIN 自己。
表结构：dwd_order order_id STRING -- 订单ID user_id STRING -- 用户ID pay_amount DECIMAL -- 支付金额 dt STRING -- 支付日期 yyyy-MM-dd
参考 SQL SELECT DISTINCT a.user_id, a.dt FROM dwd_order a JOIN dwd_order b ON a.user_id = b.user_id AND a.dt = b.dt AND a.order_id < b.order_id -- 避免自身配对和重复配对 -- 或更简洁的写法： SELECT user_id, dt FROM dwd_order GROUP BY user_id, dt HAVING COUNT(*) >= 2
不等值 JOIN：价格区间匹配
找出价格在某个区间范围内的匹配商品。不等值 JOIN 使用 BETWEEN 或 >/< 代替 = 做关联。
表结构 products: product_id, product_name, price price_tiers: tier_name, min_price, max_price
参考 SQL SELECT p.product_id, p.product_name, p.price, t.tier_name FROM products p JOIN price_tiers t ON p.price >= t.min_price AND p.price < t.max_price
注意：不等值 JOIN 无法走 Hash Join 优化，大表关联性能较差，建议先过滤缩小数据量。
LATERAL VIEW EXPLODE：Hive 行转列 进阶
一个订单包含多个商品 ID（以数组存储），需要将每个商品 ID 拆成独立行进行分析。
表结构：ods_order order_id STRING -- 订单ID product_ids ARRAY<STRING> -- 商品ID数组 ["P001","P002","P003"] pay_amount DECIMAL -- 总支付金额
参考 SQL（HiveSQL） SELECT order_id, pid, pay_amount FROM ods_order LATERAL VIEW EXPLODE(product_ids) tmp AS pid -- 一个含 3 个商品的订单会被拆成 3 行 -- 每行的 pid 分别对应数组中的一个元素
EXISTS vs IN：性能差异与适用场景
对比维度INEXISTS
工作原理先执行子查询生成结果集，再逐行匹配对外层每行执行子查询，找到即返回
子查询结果集大慢（需加载整个结果集）快（找到一条即停止）
外层表大快（子查询只执行一次）慢（每行都触发子查询）
NULL 处理IN 遇到 NULL 会有问题EXISTS 不受 NULL 影响
经验法则子查询结果集小时用 IN子查询结果集大时用 EXISTS
等价改写示例 -- IN 写法 SELECT * FROM users WHERE user_id IN (SELECT user_id FROM orders) -- EXISTS 写法（大表场景更快） SELECT * FROM users u WHERE EXISTS ( SELECT 1 FROM orders o WHERE o.user_id = u.user_id )

四、日期函数实战（HiveSQL / SparkSQL） 电商数据处理高频使用
常用日期函数速查 必背
函数说明示例
date_format日期格式化date_format("2024-03-15", "yyyyMM") → "202403"
date_add日期加 N 天date_add("2024-03-15", 7) → "2024-03-22"
date_sub日期减 N 天date_sub("2024-03-15", 1) → "2024-03-14"
datediff两日期相差天数datediff("2024-03-15", "2024-03-01") → 14
months_between两日期相差月数months_between("2024-06-15", "2024-03-15") → 3.0
year / month / day提取年/月/日year("2024-03-15") → 2024
last_day当月最后一天last_day("2024-02-10") → "2024-02-29"
next_day下一个周几next_day("2024-03-13", "MO") → "2024-03-18"
实用日期计算技巧
取本周一 date_sub( next_day(current_date(), "MO"), 7 )
取上月最后一天 date_sub( trunc(current_date(), "MM"), -- 本月第一天 1 -- 减一天 = 上月最后一天 )
取本季度第一天 trunc(current_date(), "quarter") -- 或手动计算： concat( year(current_date()), "-", lpad((floor((month(current_date())-1)/3)*3+1), 2, "0"), "-01" )
时区处理：UTC 转北京时间
跨境电商数据通常以 UTC 时间戳存储，分析时需转换为本地时区。
HiveSQL 时区转换 -- 方式一：from_utc_timestamp SELECT from_utc_timestamp(event_time, "Asia/Shanghai") AS beijing_time FROM events -- 方式二：手动加 8 小时（简单但不处理夏令时） SELECT date_add(event_time, 8.0/24) AS beijing_time
SparkSQL 时区转换 SELECT from_utc_timestamp(timestamp_col, "Asia/Shanghai") AS cn_time, from_utc_timestamp(timestamp_col, "America/Los_Angeles") AS us_time, from_utc_timestamp(timestamp_col, "Europe/London") AS uk_time FROM events
跨境电商场景：统计各时区的 GMV
不同站点的用户处于不同时区，需要按各自本地日期统计 GMV。
完整 SQL SELECT r.timezone, date_format( from_utc_timestamp(o.pay_time, r.timezone), "yyyy-MM-dd" ) AS local_date, SUM(o.pay_amount) AS gmv FROM dwd_order o JOIN dim_region r ON o.region_id = r.region_id GROUP BY r.timezone, date_format( from_utc_timestamp(o.pay_time, r.timezone), "yyyy-MM-dd" ) ORDER BY r.timezone, local_date

五、行列转换 报表开发必备技能
行转列：一行拆多行
将数组、Map 等复杂类型字段拆展为多行，是 Hive 数据处理的常见操作。
Hive：LATERAL VIEW EXPLODE -- 拆数组 SELECT order_id, pid FROM orders LATERAL VIEW EXPLODE(product_ids) tmp AS pid -- 拆 Map SELECT order_id, attr_key, attr_value FROM orders LATERAL VIEW EXPLODE(attributes) tmp AS attr_key, attr_value
Presto / Trino：UNNEST SELECT order_id, pid FROM orders CROSS JOIN UNNEST(product_ids) AS t(pid)
列转行：多行合一行
将分散在多行的数据，聚合为一行多列的报表格式。核心技巧是 CASE WHEN + GROUP BY。
示例：将月度指标从多行转为一行多列 -- 原始数据：每月一行 -- | month | metric | value | -- | 01 | gmv | 10000 | -- | 01 | uv | 5000 | -- | 01 | orders | 200 | SELECT month, MAX(CASE WHEN metric = "gmv" THEN value END) AS gmv, MAX(CASE WHEN metric = "uv" THEN value END) AS uv, MAX(CASE WHEN metric = "orders" THEN value END) AS orders FROM monthly_metrics GROUP BY month -- 结果： -- | month | gmv | uv | orders | -- | 01 | 10000 | 5000 | 200 |
实战：多行变一行多列的月度报表
电商场景中，需要将每个类目每月的 GMV 展示为类目为行、月份为列的报表。
完整 SQL SELECT category, SUM(CASE WHEN mon = "01" THEN gmv ELSE 0 END) AS jan_gmv, SUM(CASE WHEN mon = "02" THEN gmv ELSE 0 END) AS feb_gmv, SUM(CASE WHEN mon = "03" THEN gmv ELSE 0 END) AS mar_gmv, SUM(CASE WHEN mon = "04" THEN gmv ELSE 0 END) AS apr_gmv, SUM(CASE WHEN mon = "05" THEN gmv ELSE 0 END) AS may_gmv, SUM(CASE WHEN mon = "06" THEN gmv ELSE 0 END) AS jun_gmv FROM ( SELECT category, date_format(dt, "MM") AS mon, SUM(pay_amount) AS gmv FROM dwd_order_detail WHERE dt >= "2024-01-01" AND dt < "2024-07-01" GROUP BY category, date_format(dt, "MM") ) t GROUP BY category ORDER BY category
Hive collect_list / collect_set 聚合
将多行值聚合为一个数组，是另一种形式的"列转行"。
示例：聚合每个用户购买的所有商品 SELECT user_id, collect_list(product_name) AS all_products, collect_set(category) AS unique_categories, concat_ws(",", collect_set(category)) AS categories_str FROM dwd_order_detail GROUP BY user_id
  • collect_list：保留重复值、保留顺序
  • collect_set：去重，类似 DISTINCT
  • concat_ws：将数组用分隔符拼接为字符串

六、实战练习题集（电商场景） 8 道经典面试真题
练习 1：统计每月 GMV、订单数、UV，计算同比环比 必练
电商数据分析最基础的需求：按月汇总核心指标，并计算环比（MoM）和同比（YoY）增长率。
表结构：dwd_order order_id STRING -- 订单ID user_id STRING -- 用户ID pay_amount DECIMAL -- 支付金额 status STRING -- 订单状态（paid/cancelled/refunded） dt STRING -- 支付日期 yyyy-MM-dd
查看参考答案
WITH monthly AS ( SELECT date_format(dt, "yyyy-MM") AS mon, SUM(pay_amount) AS gmv, COUNT(DISTINCT order_id) AS order_cnt, COUNT(DISTINCT user_id) AS uv FROM dwd_order WHERE status = "paid" GROUP BY date_format(dt, "yyyy-MM") ) SELECT mon, gmv, order_cnt, uv, -- 环比（与上月对比） LAG(gmv, 1) OVER (ORDER BY mon) AS last_month_gmv, ROUND((gmv - LAG(gmv, 1) OVER (ORDER BY mon)) / LAG(gmv, 1) OVER (ORDER BY mon) * 100, 2) AS gmv_mom_pct, -- 同比（与去年同月对比） LAG(gmv, 12) OVER (ORDER BY mon) AS last_year_gmv, ROUND((gmv - LAG(gmv, 12) OVER (ORDER BY mon)) / LAG(gmv, 12) OVER (ORDER BY mon) * 100, 2) AS gmv_yoy_pct FROM monthly ORDER BY mon
练习 2：找出连续 3 天都有购买的用户 必练
经典的"连续登录 / 连续签到"问题变体。核心思路：日期减去行号得到分组标记，同组内连续天数 >= 3 的即为目标用户。
表结构：dwd_order（同上） user_id, dt（取去重后的购买日期）
查看参考答案
WITH user_dates AS ( -- 每个用户每天只保留一条 SELECT DISTINCT user_id, dt FROM dwd_order WHERE status = "paid" ), numbered AS ( SELECT user_id, dt, -- 日期减去行号 = 如果连续则差值相同 date_sub(dt, ROW_NUMBER() OVER ( PARTITION BY user_id ORDER BY dt )) AS grp FROM user_dates ), consecutive AS ( SELECT user_id, grp, COUNT(*) AS consecutive_days, MIN(dt) AS start_date, MAX(dt) AS end_date FROM numbered GROUP BY user_id, grp HAVING COUNT(*) >= 3 ) SELECT DISTINCT user_id FROM consecutive -- 如果需要看具体连续区间： -- SELECT user_id, start_date, end_date, consecutive_days -- FROM consecutive
核心思路：连续日期减去递增行号 = 相同常量。例如：03-01 减 1 = 02-28，03-02 减 2 = 02-28，03-03 减 3 = 02-28。三天的 grp 值相同，说明连续。
练习 3：每个商品类目的 Top 3 热销商品 核心
分组取 TopN 是面试最高频的窗口函数题。核心用 ROW_NUMBER + PARTITION BY。
表结构：dwd_order_detail order_id STRING -- 订单ID product_id STRING -- 商品ID product_name STRING -- 商品名称 category STRING -- 商品类目 pay_amount DECIMAL -- 支付金额 dt STRING -- 日期
查看参考答案
WITH product_sales AS ( SELECT category, product_id, product_name, SUM(pay_amount) AS total_sales, COUNT(*) AS order_cnt FROM dwd_order_detail WHERE dt >= "2024-01-01" GROUP BY category, product_id, product_name ), ranked AS ( SELECT *, ROW_NUMBER() OVER ( PARTITION BY category ORDER BY total_sales DESC ) AS rn FROM product_sales ) SELECT category, rn AS rank, product_id, product_name, total_sales, order_cnt FROM ranked WHERE rn <= 3 ORDER BY category, rn
练习 4：RFM 用户分层 进阶
按最近购买时间（R）、购买频次（F）、消费金额（M）三个维度给用户打标签，划分为 8 个群体。
表结构：dwd_order user_id, pay_amount, status, dt
查看参考答案
WITH rfm_base AS ( SELECT user_id, datediff("2024-06-30", MAX(dt)) AS recency, -- R: 距最近一次购买的天数 COUNT(DISTINCT order_id) AS frequency, -- F: 购买次数 SUM(pay_amount) AS monetary -- M: 累计消费金额 FROM dwd_order WHERE status = "paid" GROUP BY user_id ), rfm_score AS ( SELECT *, -- 用 NTILE 将每个维度分为高/低两档 CASE WHEN NTILE(2) OVER (ORDER BY recency ASC) = 1 THEN "高" ELSE "低" END AS r_level, -- recency 越小越好 CASE WHEN NTILE(2) OVER (ORDER BY frequency DESC) = 1 THEN "高" ELSE "低" END AS f_level, CASE WHEN NTILE(2) OVER (ORDER BY monetary DESC) = 1 THEN "高" ELSE "低" END AS m_level FROM rfm_base ) SELECT user_id, recency, frequency, monetary, r_level, f_level, m_level, CASE WHEN r_level="高" AND f_level="高" AND m_level="高" THEN "重要价值用户" WHEN r_level="高" AND f_level="低" AND m_level="高" THEN "重要发展用户" WHEN r_level="低" AND f_level="高" AND m_level="高" THEN "重要保持用户" WHEN r_level="低" AND f_level="低" AND m_level="高" THEN "重要挽留用户" WHEN r_level="高" AND f_level="高" AND m_level="低" THEN "一般价值用户" WHEN r_level="高" AND f_level="低" AND m_level="低" THEN "一般发展用户" WHEN r_level="低" AND f_level="高" AND m_level="低" THEN "一般保持用户" WHEN r_level="低" AND f_level="低" AND m_level="低" THEN "流失用户" END AS user_tag FROM rfm_score
练习 5：留存率计算（7日 / 30日） 必练
统计每天新注册用户在第 1 天、第 7 天、第 30 天是否回访，计算留存率。
表结构 dim_user: user_id, register_date dwd_user_active: user_id, active_date
查看参考答案
WITH new_users AS ( SELECT user_id, register_date FROM dim_user WHERE register_date >= "2024-01-01" ), retention AS ( SELECT n.register_date, COUNT(DISTINCT n.user_id) AS new_user_cnt, COUNT(DISTINCT CASE WHEN datediff(a.active_date, n.register_date) = 1 THEN n.user_id END) AS day1_retained, COUNT(DISTINCT CASE WHEN datediff(a.active_date, n.register_date) = 7 THEN n.user_id END) AS day7_retained, COUNT(DISTINCT CASE WHEN datediff(a.active_date, n.register_date) = 30 THEN n.user_id END) AS day30_retained FROM new_users n LEFT JOIN dwd_user_active a ON n.user_id = a.user_id GROUP BY n.register_date ) SELECT register_date, new_user_cnt, ROUND(day1_retained / new_user_cnt * 100, 2) AS day1_retention_rate, ROUND(day7_retained / new_user_cnt * 100, 2) AS day7_retention_rate, ROUND(day30_retained / new_user_cnt * 100, 2) AS day30_retention_rate FROM retention ORDER BY register_date
练习 6：漏斗分析（浏览→加购→下单→支付） 核心
计算从浏览到支付各环节的转化率和流失率，找出最大断层环节。
表结构：dwd_user_event user_id STRING -- 用户ID event_type STRING -- 事件类型：view / add_cart / create_order / pay dt STRING -- 事件日期
查看参考答案
WITH funnel AS ( SELECT COUNT(DISTINCT CASE WHEN event_type = "view" THEN user_id END) AS view_uv, COUNT(DISTINCT CASE WHEN event_type = "add_cart" THEN user_id END) AS cart_uv, COUNT(DISTINCT CASE WHEN event_type = "create_order" THEN user_id END) AS order_uv, COUNT(DISTINCT CASE WHEN event_type = "pay" THEN user_id END) AS pay_uv FROM dwd_user_event WHERE dt = "2024-03-15" ) SELECT view_uv, cart_uv, order_uv, pay_uv, -- 各环节转化率（相对上一步） ROUND(cart_uv / view_uv * 100, 2) AS view_to_cart_pct, ROUND(order_uv / cart_uv * 100, 2) AS cart_to_order_pct, ROUND(pay_uv / order_uv * 100, 2) AS order_to_pay_pct, -- 整体转化率（相对第一步） ROUND(pay_uv / view_uv * 100, 2) AS overall_cvr FROM funnel
练习 7：每周 GMV 的周环比增长率
按自然周统计 GMV，计算周环比（WoW = Week over Week）增长率。
表结构：dwd_order（同上） order_id, pay_amount, status, dt
查看参考答案
WITH weekly AS ( SELECT -- Hive: weekofyear(); SparkSQL: weekofyear() year(dt) AS yr, weekofyear(dt) AS wk, MIN(dt) AS week_start, SUM(pay_amount) AS weekly_gmv FROM dwd_order WHERE status = "paid" GROUP BY year(dt), weekofyear(dt) ) SELECT yr, wk, week_start, weekly_gmv, LAG(weekly_gmv, 1) OVER (ORDER BY yr, wk) AS last_week_gmv, ROUND( (weekly_gmv - LAG(weekly_gmv, 1) OVER (ORDER BY yr, wk)) / LAG(weekly_gmv, 1) OVER (ORDER BY yr, wk) * 100, 2 ) AS wow_growth_pct FROM weekly ORDER BY yr, wk
练习 8：商品关联分析（购物篮分析） 进阶
找出经常一起购买的商品对，用于推荐系统的"猜你喜欢"和交叉销售。

WITH order_products AS ( SELECT DISTINCT order_id, product_id, product_name FROM dwd_order_detail ) SELECT a.product_name AS product_a, b.product_name AS product_b, COUNT(DISTINCT a.order_id) AS co_buy_orders, -- 支持度 = 同时购买的订单数 / 总订单数 ROUND( COUNT(DISTINCT a.order_id) / (SELECT COUNT(DISTINCT order_id) FROM dwd_order_detail) * 100, 4 ) AS support_pct FROM order_products a JOIN order_products b ON a.order_id = b.order_id AND a.product_id < b.product_id -- 避免重复配对和自配对 GROUP BY a.product_name, b.product_name HAVING COUNT(DISTINCT a.order_id) >= 10 -- 过滤低频组合 ORDER BY co_buy_orders DESC LIMIT 50
结果说明：co_buy_orders 越高说明两个商品越常被一起购买。support_pct 是关联规则中的"支持度"指标。可进一步计算"置信度"和"提升度"做更精细的分析。

八、SQL 性能优化原则 写出高效的大数据 SQL
分区裁剪 必背
Hive / SparkSQL 表通常按日期分区存储。WHERE 中包含分区字段可以跳过无关分区的数据扫描，减少 IO 数量级。
正确做法：WHERE 包含分区字段 SELECT * FROM dwd_order WHERE dt >= "2024-01-01" AND dt < "2024-02-01" -- 只读取 1 月分区的数据文件
错误做法：对分区字段做函数计算 SELECT * FROM dwd_order WHERE date_format(dt, "yyyy-MM") = "2024-01" -- 分区裁剪失效！引擎无法推断需要哪些分区 -- 会扫描全部分区数据
谓词下推：过滤条件尽量提前
将 WHERE 过滤条件尽量下推到数据读取阶段，减少参与 JOIN / 聚合的数据量。
优化前（先 JOIN 再过滤，浪费资源） SELECT * FROM orders o JOIN users u ON o.user_id = u.user_id WHERE o.dt = "2024-03-15"
优化后（先过滤再 JOIN，数据量大幅减少） SELECT * FROM (SELECT * FROM orders WHERE dt = "2024-03-15") o JOIN users u ON o.user_id = u.user_id -- 现代引擎通常会自动做谓词下推 -- 但复杂 SQL 中引擎可能优化失败，手动改写更保险
避免 SELECT * & 小表广播 JOIN
  • 避免 SELECT *：列式存储（Parquet/ORC）只读取需要的列。SELECT * 会读取全部列，浪费大量 IO
  • 只查需要的列：SELECT user_id, pay_amount 比 SELECT * 快数倍
  • 小表广播 JOIN：当一侧表很小（< 几十 MB），广播到每个 Executor 避免 Shuffle
Hive 广播 JOIN（MapJoin） SET hive.auto.convert.join = true; SET hive.mapjoin.smalltable.filesize = 25000000; -- 25MB SELECT /*+ MAPJOIN(dim) */ * FROM fact_orders fact JOIN dim_region dim ON fact.region_id = dim.region_id
SparkSQL 广播 JOIN SELECT /*+ BROADCAST(dim) */ * FROM fact_orders fact JOIN dim_region dim ON fact.region_id = dim.region_id
数据倾斜处理 进阶
当某些 Key 的数据量远超其他 Key 时，会导致个别 Task 耗时极长。
  • NULL 值处理：将 NULL Key 替换为随机值分散处理
  • 热点 Key 加盐：给倾斜 Key 加随机前缀，分散到多个 Reducer
  • 两阶段聚合：先局部聚合（加盐 + GROUP BY），再全局聚合（去盐 + GROUP BY）
  • AQE 自动处理：Spark 3.x 开启 spark.sql.adaptive.enabled = true
NULL Key 分散处理 SELECT COALESCE(user_id, CONCAT("null_", CAST(RAND() * 100 AS INT))) AS join_key, * FROM orders
两阶段聚合（先加盐聚合，再去盐聚合） -- 第一阶段：加盐打散 WITH salted AS ( SELECT CONCAT(category, "_", CAST(FLOOR(RAND()*10) AS INT)) AS salted_key, category, pay_amount FROM orders ) , partial AS ( SELECT salted_key, category, SUM(pay_amount) AS partial_sum FROM salted GROUP BY salted_key, category ) -- 第二阶段：去盐汇总 SELECT category, SUM(partial_sum) AS total FROM partial GROUP BY category

EXPLAIN 查看执行计划
优化 SQL 前，先用 EXPLAIN 查看执行计划，了解引擎如何执行你的查询。
查看执行计划 EXPLAIN SELECT user_id, SUM(pay_amount) FROM dwd_order WHERE dt = "2024-03-15" GROUP BY user_id -- 重点关注： -- 1. 是否命中分区裁剪（Partition pruning） -- 2. JOIN 类型（BroadcastHashJoin / SortMergeJoin / ShuffleHashJoin） -- 3. 是否有全表扫描（TableScan without filter） -- 4. Shuffle 数据量大小
