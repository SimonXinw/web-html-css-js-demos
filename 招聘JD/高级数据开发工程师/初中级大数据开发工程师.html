一、分布式计算引擎：Spark & Flink 40 分 · 核心引擎
Q1: 详述 Spark 的 DAG 执行原理 — RDD、Stage、Task 是什么关系？宽依赖 vs 窄依赖对 Shuffle 有何影响？
Spark 核心

考察关键点

  • DAG 划分逻辑：遇到宽依赖（如 groupByKey、join）时 DAG 切分 Stage；窄依赖（map、filter）在同 Stage 内流水线执行，无需 Shuffle。
  • 宽/窄依赖本质：窄依赖 = 父 RDD 的一个分区只被子 RDD 的一个分区使用；宽依赖 = 一个父分区被多个子分区依赖，必须等父 Stage 全部完成才能启动子 Stage。
  • Shuffle 写放大：Shuffle Write 将数据写磁盘再通过网络传输，是 Spark 最大的性能开销；宽依赖越多，Shuffle 越重。
  • Task 粒度：一个 Stage 中，每个 RDD 分区对应一个 Task；分区数决定并行度。

评分标准 (10分)

8–10 分
能画出 DAG Stage 划分图；清楚宽/窄依赖的分区映射关系；解释 Shuffle Write/Read 的磁盘 IO 开销；提到用 reduceByKey 替代 groupByKey 减少 Shuffle 数据量。
4–7 分
知道 RDD/Stage/Task 的层级关系，知道 Shuffle 有开销，但宽/窄依赖的定义模糊。
0–3 分
分不清 Stage 和 Task；不知道 Shuffle 是什么。
查看参考答案

DAG 执行流程：

  1. Driver 将程序转换为 DAG（有向无环图），节点是 RDD，边是依赖关系
  2. 遇到宽依赖（Shuffle 边界）切分 Stage；Stage 内全是窄依赖，可流水线执行
  3. 每个 Stage 分解为多个 Task（= 分区数）提交给 Executor 并行执行

关键优化：reduceByKey vs groupByKey

// groupByKey：先 Shuffle 全量数据，再聚合 → 网络传输量大
rdd.groupByKey().mapValues(_.sum)

// reduceByKey：先在 Map 端局部聚合（Combiner），再 Shuffle → 数据量减少 90%+
rdd.reduceByKey(_ + _)  // 推荐
Q2: Spark 中 RDD、DataFrame、Dataset 有什么区别？什么是"懒执行（Lazy Evaluation）"？Action 和 Transformation 如何区分？
Spark 基础

考察关键点

  • RDD：最底层的分布式数据集，无 Schema，需要手写序列化，类型安全但开发效率低。
  • DataFrame：带 Schema 的分布式表，类似 SQL 表，有 Catalyst 查询优化器加持，推荐日常使用。
  • Dataset：DataFrame 的强类型版本（Scala/Java），兼顾编译期类型安全和查询优化，Python 中等同 DataFrame。
  • 懒执行（Lazy Evaluation）：Transformation（如 map/filter/groupBy）只记录操作计划，不立即执行；只有遇到 Action（如 count/show/write）时才真正触发计算，好处是 Spark 可以整体优化执行计划。
  • Transformation vs Action：Transformation 返回新 RDD/DataFrame（懒执行）；Action 返回结果或写入外部存储（立即执行）。

评分标准 (10分)

8–10 分
清楚三者的演进关系（RDD → DataFrame → Dataset）；能举例说明 Lazy Evaluation 的好处（整体优化）；能列出常见 Transformation 和 Action 各 3 个以上。
4–7 分
知道 DataFrame 比 RDD 好用，知道懒执行的大概含义，但 Transformation/Action 区分不够清晰。
0–3 分
不清楚三者区别，或不知道懒执行是什么。
查看参考答案

常见操作分类：

类型常见操作说明
Transformation（懒）map, filter, groupBy, join, select, withColumn返回新 DataFrame，不触发计算
Action（立即执行）count, show, collect, write, first, take触发整个计划执行，返回结果
 
// 整个链条都是 Transformation，不会真正执行
val result = df
  .filter($"status" === "paid")   // Transformation
  .groupBy("user_id")             // Transformation
  .agg(sum("amount"))             // Transformation

// 直到这里触发 Action，才真正提交 Spark Job
result.show()  // Action → 触发计算！
Q3: Kafka 是什么？Topic、Partition、消费者组（Consumer Group）分别是什么概念？消费者为什么不能超过分区数？
Kafka 基础

考察关键点

  • Kafka 定位：高吞吐、持久化的分布式消息队列，常用作大数据的"数据总线"——采集系统的数据写入 Kafka，计算系统（Spark/Flink）从 Kafka 消费。
  • Topic：消息的逻辑分类，类似"频道"。不同业务数据放不同 Topic（如 order_topic、click_topic）。
  • Partition（分区）：Topic 的物理分片，实现并行读写；同一 Partition 内消息有序；多个 Partition 分布在不同 Broker，实现横向扩展。
  • Consumer Group（消费者组）：同一 Group 内的消费者共同消费一个 Topic，每个 Partition 只被 Group 内的一个消费者消费；不同 Group 相互独立，各自消费全量数据。
  • 消费者不能超过分区数的原因：一个 Partition 最多分配给 Group 内一个 Consumer，多余的 Consumer 会空闲，不能提升并行度。

评分标准 (10分)

8–10 分
能清楚解释 Topic/Partition/Consumer Group 三者关系；理解多消费者组可以各自独立消费（如实时计算和离线备份同时消费同一 Topic）；知道 offset 是消费位点。
4–7 分
知道 Kafka 是消息队列，理解 Topic 和 Partition 的基本概念，但 Consumer Group 的语义解释不清楚。
0–3 分
不了解 Kafka，或只知道它是"消息队列"，无法解释核心概念。
查看参考答案

Consumer Group 的两大特性：

  • 组内竞争消费：同 Group 的 Consumer 瓜分所有 Partition，每条消息只被消费一次 → 用于水平扩展消费能力
  • 组间广播：不同 Group 相互独立，都能消费全量数据 → 用于同一数据多路消费（实时计算 + 落库备份）

Q4: Spark Streaming 和 Flink 的核心架构差异？什么场景选 Flink，什么场景选 Spark？
技术选型

考察关键点

  • 计算模型差异：Spark Streaming 是微批（Micro-Batch），将流数据切成小批次处理，延迟 100ms–秒级；Flink 是真正的流处理（逐条处理），延迟可达毫秒级。
  • 状态管理：Flink 原生支持有状态流计算，状态存储在 State Backend；Spark Streaming 状态管理较弱（Stateful Transformation 有限制）。
  • 选 Flink 的场景：毫秒级延迟需求、复杂事件处理（CEP）、长周期状态聚合（如 session window）、需要 Exactly-Once 的金融场景。
  • 选 Spark 的场景：批流一体（Spark 3.x Structured Streaming）、团队已有 Spark 技术积累、机器学习 pipeline 集成（MLlib）、SQL 分析为主。

评分标准 (10分)

8–10 分
清晰区分微批 vs 真流处理的延迟差异；能给出具体业务场景的选型理由；提到 Spark 3.x Structured Streaming 已支持连续处理模式缩短延迟。
4–7 分
知道 Flink 延迟低、Spark 生态强，但具体场景判断不够细。
0–3 分
认为 Flink 全面优于 Spark，或反之，无法给出场景化判断。
二、数据仓库设计与建模 30 分 · 分层 · 模型
Q5: 为什么数据仓库要分层（ODS/DWD/DWS/ADS）？每层的职责是什么？不分层有什么问题？
数仓分层

考察关键点

  • 各层职责：ODS（贴源层，保持原始数据格式，做轻量清洗）→ DWD（明细层，数据清洗/标准化/关联维表，宽表）→ DWS（汇总层，按主题聚合，轻度汇总）→ ADS（应用层，面向具体指标/报表）。
  • 分层的价值：数据复用（DWS 层被多个 ADS 共享，避免重复计算）；问题隔离（数据质量问题定位到具体层）；血缘清晰；灵活应对需求变化。
  • 不分层的问题：业务逻辑直接堆在 SQL 里，牵一发动全身；大量重复计算导致资源浪费；数据问题难以溯源。
  • 维度建模：星型模式（事实表 + 维度表）vs 雪花模式（维度表进一步规范化）；电商场景通常用星型，查询性能更好。

评分标准 (10分)

8–10 分
能给出具体数据流举例（订单数据从 ODS 到 ADS 的流转）；指出不分层的具体痛点；提到缓慢变化维（SCD）的处理方式。
4–7 分
能说出各层名称和基本作用，但对分层价值的理解停留在"规范"层面，缺乏实际问题意识。
0–3 分
不清楚各层区别，ODS 和 DWD 混淆。

Q6: 什么是缓慢变化维（SCD）？电商场景中商品价格变更历史，应该用哪种 SCD 方式处理？
维度建模

考察关键点

  • SCD Type 1：直接覆盖旧值，无历史记录；适合错误数据的修正，不适合需要追溯历史的场景。
  • SCD Type 2：新增一行记录新值，旧行标记 end_date 或 is_current=false；保留完整历史；订单分析"下单时的商品价格"必须用 Type 2。
  • SCD Type 3：在同一行增加 prev_value 字段记录上一个值；只保留一次历史变化，代价小但灵活性差。
  • 拉链表：SCD Type 2 的工程实现；用 start_dt / end_dt 记录有效期，end_dt = '9999-12-31' 表示当前有效；查询某时刻数据用时间范围过滤。

评分标准 (10分)

8–10 分
能区分三种 SCD 的适用场景；清楚拉链表的实现方式和查询方式；提到拉链表会导致数据膨胀，需要按时间分区裁剪。
4–7 分
知道 SCD 概念，能说出 Type 1/2/3，但不清楚拉链表的具体 SQL 实现。
0–3 分
不知道 SCD，直接覆盖历史数据。
查看参考答案：拉链表实现 
-- 商品价格拉链表结构（SCD Type 2）
CREATE TABLE dim_product_price_zip (
  product_id  STRING,
  price       DECIMAL(10,2),
  start_dt    STRING,  -- 价格生效日期
  end_dt      STRING   -- 价格失效日期，'9999-12-31' 表示当前有效
);

-- 查询某订单下单时（2024-11-11）的商品价格
SELECT p.price
FROM dwd_order_detail o
JOIN dim_product_price_zip p
  ON o.product_id = p.product_id
  AND o.order_date BETWEEN p.start_dt AND p.end_dt;

注意：拉链表数据量会随时间增大，建议按 start_dt 分区 + 定期压缩历史。

Q7: 电商大宽表（事实表）设计时，如何权衡冗余字段存储成本 vs 查询性能？什么情况下"反范式化"是合理的？
宽表设计

考察关键点

  • 范式化 vs 反范式化：关系型 OLTP 追求范式化（减少冗余）；OLAP 数仓追求反范式化（减少 Join），用存储换查询速度。
  • 冗余合理性判断：频繁被查询的维度字段（如用户城市、商品类目）值得冗余到事实表，避免高频 Join 扫描大维表。
  • 冗余代价：维度值变更时需要同步更新宽表（或接受历史冗余值），维护成本高；列式存储（如 Parquet/ORC）对重复值有高压缩率，存储代价可控。
  • Bitmap/字典压缩：ClickHouse 等 OLAP 引擎对低基数（low cardinality）字符串列自动字典编码，冗余对存储影响有限。

评分标准 (10分)

8–10 分
能量化说明宽表对 Join 次数的减少；提到列存压缩降低冗余代价；讨论冗余字段的更新维护策略（快照 vs 拉链）。
4–7 分
知道数仓用宽表提高查询性能，但对权衡的量化分析不够深入。
0–3 分
数仓设计仍然按 OLTP 范式，大量 Join 导致查询极慢。

Q8: Kafka 如何保证消息不丢失、不重复？分区（Partition）、消费组（Consumer Group）的设计原则是什么？
Kafka 核心

考察关键点

  • 不丢失（Producer 端）：acks=all 要求 Leader + 所有 ISR 副本确认后才返回成功；retries > 0 自动重试；结合幂等性（enable.idempotence=true）避免重试导致重复。
  • 不丢失（Consumer 端）：关闭自动提交 offset（enable.auto.commit=false），业务逻辑处理完后手动提交；若先提交再处理，崩溃会丢消息。
  • 幂等性 vs 事务：幂等性保证单 Producer 单 Session 内不重复；事务（transactional.id）保证跨 Topic/跨 Partition 的原子写。
  • 分区设计原则：分区数 = 消费者并发数（一个 Partition 同时只能被一个消费组内的一个消费者消费）；分区过多增加元数据开销；有序性要求用同一分区键（如 userId）路由同一分区。

评分标准 (10分)

8–10 分
能区分 acks=0/1/all 三种级别的语义；解释幂等性的 PID+序列号机制；知道消费者数量超过分区数时多余消费者空闲；提到 Kafka 事务的使用场景。
4–7 分
知道 acks=all 最安全，手动提交 offset，但幂等性和事务细节不清楚。
0–3 分
认为 Kafka 默认就是 Exactly-Once，不知道 acks 参数的作用。
查看参考答案 
# Producer 不丢失配置
props.put("acks", "all");
props.put("retries", 3);
props.put("enable.idempotence", "true");   // 幂等：Producer 重启不重复
props.put("transactional.id", "tx-001");   // 事务：跨 Topic 原子写

# Consumer 不丢失配置
props.put("enable.auto.commit", "false");  // 关闭自动提交

// 消费逻辑：先处理，后提交（处理完才算消费成功）
ConsumerRecords records = consumer.poll(Duration.ofMillis(100));
processRecords(records);   // 1. 先处理
consumer.commitSync();     // 2. 再提交 offset

分区数设置建议：分区数 = max(期望吞吐 / 单分区吞吐上限, 消费者最大并发数)；后期可增加分区，但会破坏基于 key 的顺序性，需评估影响。

Q9: ClickHouse、Apache Doris、Presto 三种 OLAP 引擎如何选型？各自适合什么场景？
OLAP 选型

考察关键点

  • ClickHouse：列存 + 向量化执行，单表超大数据量（万亿行）点查和聚合极快；但 Join 性能弱（需要用字典表/物化视图优化）；适合日志/埋点等宽表分析。
  • Doris（StarRocks）：MPP 架构，Join 性能强，支持高并发点查；同时支持明细查询和聚合查询；适合多表关联分析、实时数仓场景。
  • Presto（Trino）：联邦查询引擎，可跨 Hive/MySQL/ES/Kafka 联合查询；无自己的存储；适合临时分析、数据探索，不适合超高 QPS。
  • 选型核心维度：数据量级、查询并发、Join 复杂度、实时性要求、运维成本。

4–7 分
知道三者的基本特点，但无法给出场景化的选型依据。

Q10: Hive 分区（Partition）和分桶（Bucket）的原理是什么？查询时如何利用分区裁剪和 Map Join 优化性能？
Hive 优化

考察关键点

  • 分区裁剪：分区字段作为目录存储（如 /dt=2024-11-11/），查询时 WHERE 条件包含分区字段则只扫描对应目录，跳过其他分区的 IO。
  • 分桶原理：按字段 Hash 取模将数据分到固定数量的桶文件；相同桶号的数据一定在同一文件；两张按相同字段分桶的表 Join 时可用 Bucket Map Join，无需全量 Shuffle。
  • Map Join（小表广播）：小表（通常 <25MB）加载到每个 Mapper 的内存 Hash 表；大表 Map 阶段直接在内存中完成 Join，无需 Reduce 阶段，避免 Shuffle。
  • 动态分区：INSERT 时根据数据内容自动创建分区，需开启 hive.exec.dynamic.partition=true；注意动态分区过多会生成大量小文件。

评分标准 (10分)

8–10 分
能说明分区裁剪的物理存储机制；解释 Bucket Map Join 的前提条件（桶数量倍数关系）；知道 Map Join 自动触发的大小阈值；提到小文件问题和 COMPACT 操作。
4–7 分
知道分区可以减少扫描量，会用 Map Join，但分桶原理不清楚。
0–3 分
不知道分区裁剪，全表扫描后再 WHERE 过滤。

Q11: SQL 实战题 — 用窗口函数计算每个用户的首次购买日期、最近购买日期、累计消费金额，以及每天的新用户数（首次下单当天算新用户）。

考察关键点

  • MIN/MAX 聚合：MIN(order_date) 取首购日期，MAX(order_date) 取最近购买日期，SUM(amount) 取累计消费。
  • 窗口函数：用 ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY order_date) 给每个用户的订单排序，取 rn=1 即首单。
  • 新用户定义：首次下单日期 = 统计日期的用户，即当天新增的消费者。
  • GROUP BY vs 窗口函数：GROUP BY 会合并行，窗口函数保留原始行数，两者选择取决于是否需要保留明细。

方法一：GROUP BY 聚合（统计每用户概况）

SELECT
  user_id,
  MIN(order_date)   AS first_order_date,    -- 首次购买日期
  MAX(order_date)   AS last_order_date,     -- 最近购买日期
  SUM(amount)       AS total_amount,        -- 累计消费金额
  COUNT(order_id)   AS order_count          -- 累计订单数
FROM orders
WHERE status = 'paid'
GROUP BY user_id;

方法二：窗口函数（保留每行明细并标注首单标志）

SELECT *,
  ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY order_date ASC) AS order_rank,
  SUM(amount)   OVER (PARTITION BY user_id ORDER BY order_date
                      ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS cumulative_amount
FROM orders
WHERE status = 'paid';

每天新用户数（首次下单当天 = 新用户）

WITH first_order AS (
  SELECT user_id, MIN(order_date) AS first_date
  FROM orders WHERE status = 'paid'
  GROUP BY user_id
)
SELECT first_date AS dt, COUNT(user_id) AS new_users
FROM first_order
GROUP BY first_date
ORDER BY dt;
Q12: 如何设计大数据任务的数据质量监控体系？数据异常时如何告警和阻断下游任务？
数据质量

考察关键点

  • 质量维度：完整性（行数/关键字段非空率）、准确性（数值范围合理性）、一致性（跨表/跨系统数据对齐）、时效性（数据是否按时到达）。
  • 检查时机：ODS 入库后检查（源数据质量）；DWD 处理后检查（ETL 逻辑正确性）；ADS 输出前检查（指标合理性，如 GMV 环比变化超 50% 告警）。
  • 阻断机制：任务调度（Airflow/DolphinScheduler）中配置前置传感器（Sensor）；数据质量任务通过则下游任务继续，失败则阻断 + 告警 + 等人工处理。
  • 工具：Apache Griffin、Great Expectations、Soda Core 等开源数据质量框架；或自建规则引擎。

评分标准 (10分)

8–10 分
能设计完整的质量检查规则体系；知道如何在调度系统中插入质量阻断节点；提到业务指标异常的统计阈值设置方法（如 3-Sigma 原则）。
4–7 分
知道要做数据质量检查，有行数/非空率的概念，但系统性设计不足。
0–3 分
没有数据质量概念，出了问题才事后排查。
Q13: 小文件问题（Small File Problem）是什么？在 HDFS 和 Hive 中如何产生、有何危害、如何治理？
工程化实践

考察关键点

  • 产生原因：Spark/Hive 写入时每个 Task 生成一个文件；动态分区 + 大量分区 = 大量小文件；Kafka 小批量消费实时写入；频繁追加写入。
  • 危害：HDFS NameNode 内存占用（每个文件约 150 字节元数据）；Map Task 数量膨胀（一个小文件一个 Mapper）；查询 IO 次数多，吞吐量低。
  • 治理方案：写入时合并（coalesce(N) 减少输出分区）；定时 Compact（SparkSQL INSERT OVERWRITE 重写合并）；Hive 开启 hive.merge.smallfiles.avgsize 自动合并；Delta Lake / Iceberg 的 OPTIMIZE 命令。

评分标准 (10分)

8–10 分
能定量说明 NameNode 内存影响；给出写入时预防和事后治理两种思路；提到 Iceberg/Delta Lake 的 OPTIMIZE 命令自动合并。
4–7 分
知道小文件有害，会用 coalesce 控制输出文件数，但不了解 NameNode 元数据影响。
0–3 分
不知道小文件问题，或认为文件多少无所谓。
查看参考方案 
-- Spark 写入时控制文件数（避免产生小文件）
df.coalesce(20)           // 减少到 20 个文件（不触发 Shuffle）
  .write
  .partitionBy("dt")
  .parquet("/warehouse/fact_orders/")

-- Hive 自动合并小文件（写入后触发）
SET hive.merge.mapfiles=true;
SET hive.merge.mapredfiles=true;
SET hive.merge.smallfiles.avgsize=256000000;  -- 目标合并到 256MB

-- Iceberg / Delta Lake 定时治理
OPTIMIZE fact_orders WHERE dt >= '2024-11-01';  -- 合并历史小文件
VACUUM fact_orders RETAIN 7 DAYS;               -- 清理旧版本文件
Q14: 压轴场景题 — 设计一套电商大促（如双十一）实时数据大屏的技术方案，要求每秒更新 GMV、订单量、转化率等核心指标。
综合架构设计

考察关键点

  • 数据链路：业务库 Binlog → Kafka（CDC，如 Debezium/Canal）→ Flink 实时计算 → Redis/ClickHouse → 大屏前端推送（WebSocket/SSE）。
  • Flink 指标计算：滚动窗口（Tumbling Window）计算分钟级 GMV；会话窗口（Session Window）计算用户转化路径；State 存储累计 GMV。
  • 高并发写入：Flink 输出到 Redis（高频更新的实时指标）+ ClickHouse（历史明细供深度查询）；不直接写 MySQL（写入压力大）。
  • 容错与准确性：Flink Checkpoint 保证 Exactly-Once；大促前进行压测（数据量放大 10x）；设置背压报警防止 Flink 处理延迟。
  • 大屏数据推送：后端 Server-Sent Events / WebSocket 每秒推送最新指标到前端；前端动画展示数值变化（数字滚动效果）。

评分标准 (10分)

8–10 分
给出完整的技术链路（CDC → Kafka → Flink → Redis → 前端）；解释窗口类型选择理由；考虑到大促流量洪峰的背压处理；提到数据对账机制（实时 vs T+1 离线对比验证）。
4–7 分
能设计出基本链路，会用 Flink 窗口计算，但高并发、容错、对账等细节不完整。
0–3 分
用定时查询 MySQL 实现实时大屏（1分钟轮询），无法支撑双十一流量。
查看参考架构方案

完整技术链路：

// 数据流向
MySQL Binlog
  → Canal/Debezium (CDC) 捕获增量
  → Kafka Topic: order_events (分区按 user_id 路由)
  → Flink Stream Job
      ├── 滚动窗口 1min → GMV / 订单量（写 Redis Hash）
      ├── 滑动窗口 5min/1min → 转化率趋势
      └── 全局累计 State → 累计 GMV（写 Redis String）
  → Redis（实时大屏，毫秒级读取）
  → ClickHouse（历史明细，支持深度分析查询）
  → WebSocket Server（推送给大屏前端，1s/次）

Flink 关键配置（大促场景）：

// 背压监控：防止处理延迟累积
env.setBufferTimeout(100);  // 降低 Buffer 延迟

// Checkpoint：每30秒一次，支持故障快速恢复
env.enableCheckpointing(30000);
env.getCheckpointConfig().setCheckpointingMode(
  CheckpointingMode.EXACTLY_ONCE
);

// 大促前压测：生产者模拟 10x 流量
KafkaProducer.send(generateHighLoadEvents(10));

数据对账：每小时将 Flink 实时指标与 T+0 离线 Spark 任务结果对比，误差 >0.1% 则触发告警，保障大促数据准确性。
