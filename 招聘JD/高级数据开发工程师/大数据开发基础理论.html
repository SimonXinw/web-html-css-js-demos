一、大数据核心概念 新手必知
大数据的 5V 特征 必背
  • Volume（体量）：数据量巨大，TB → PB → EB 级别，无法用单机存储
  • Velocity（速度）：数据产生速度极快，需要实时或近实时处理
  • Variety（多样性）：结构化（数据库）、半结构化（JSON/日志）、非结构化（图片/视频）
  • Veracity（真实性）：数据质量参差不齐，有噪声、重复、缺失
  • Value（价值）：价值密度低，需要从海量数据中挖掘有用信息
分布式系统三大挑战
  • 一致性 (Consistency)：所有节点同一时刻看到的数据相同
  • 可用性 (Availability)：每个请求都能得到响应（不论成功/失败）
  • 分区容错 (Partition Tolerance)：网络分区时系统仍能继续运行
CAP 定理 三者只能同时满足其中两个。
大数据系统通常选 AP（可用 + 分区容错），牺牲强一致性，用最终一致性替代。
批处理 vs 流处理
维度批处理流处理
代表工具Hive / SparkFlink / Kafka Streams
延迟分钟～小时毫秒～秒级
数据范围有界（历史全量）无界（实时流入）
典型场景每日报表、T+1 分析实时大屏、风控告警
容错成本低（重跑即可）高（需 Checkpoint）
MapReduce 原理（基础思想）
Google 提出的分布式计算模型，是 Hadoop 的核心计算引擎，也是理解 Spark 的基础。
原始数据 → Map（切片并行处理） → Shuffle（按 Key 排序分发） → Reduce（合并聚合） → 输出结果
例：统计词频 → Map 拆分每个词输出 (word, 1)，Reduce 对相同 word 的 1 求和

二、数据存储基础 理解数据的组织方式
HDFS 分布式文件系统 核心
  • Block（数据块）：文件切分为 128MB 的 Block，分散存储在各节点
  • NameNode（主节点）：管理文件系统元数据（文件路径、Block 位置），不存实际数据
  • DataNode（数据节点）：实际存储数据块，默认 3 副本保证高可用
  • 副本策略：第 1 副本在本机，第 2 副本在同机架另一台，第 3 副本在另一机架
NameNode 是 HDFS 的单点，内存有限（每个文件约 150 Byte 元数据）
→ 大量小文件会拖垮 NameNode，这是小文件问题的根源
常见数据文件格式对比
格式特点适用场景
CSV / JSON行式存储，人类可读原始数据接入（ODS层）
Parquet列式存储，高压缩，支持谓词下推OLAP 查询首选
ORC列式存储，Hive 优化，支持 ACIDHive 仓库中间层
Avro行式+Schema 演化，序列化效率高Kafka 消息、流处理
列式存储为什么快？
行式存储：同一行的字段紧挨存储，适合查询整行（如 MySQL 的增删改查）。

列式存储：同一列的值紧挨存储，查询某几列时只需读取对应列文件，跳过无关列。
场景：SELECT SUM(price) FROM orders
行式：读取每一行所有字段 → 只取 price → 浪费大量 IO
列式：直接读取 price 列文件 → IO 减少 90%+
数据压缩格式
  • Snappy：压缩速度极快，压缩率中等，Spark 默认推荐，不支持分割
  • Gzip：压缩率高，但压缩/解压慢，适合冷数据归档
  • LZO：支持分割（可并行解压），适合大文件 MapReduce 输入
  • Zstd：新一代，兼顾高压缩率和速度，ClickHouse/Parquet 常用
压缩选择原则：实时/频繁读写 → Snappy；存储优先/冷数据 → Gzip/Zstd

三、SQL 核心知识 大数据开发日常 80% 工作量
JOIN 类型全解 必背
JOIN 类型结果说明场景
INNER JOIN两表都有匹配才保留关联确定存在的维度
LEFT JOIN左表全保留，右表无匹配为 NULL保留左表全量（最常用）
RIGHT JOIN右表全保留（少用，可转为 LEFT）-
FULL OUTER JOIN两表都保留，无匹配为 NULL数据对账、差异分析
CROSS JOIN笛卡尔积（行数相乘）日期序列扩展等特殊场景
LEFT ANTI JOIN左表中右表无匹配的行找"未购买用户"等差集
窗口函数详解 必背
窗口函数在不减少行数的情况下计算聚合值，是数据分析的核心工具。
语法结构 函数名() OVER (PARTITION BY 分组列 ORDER BY 排序列 ROWS/RANGE 窗口范围)
  • ROW_NUMBER()：序号（1,2,3...），相同值也不并列
  • RANK()：排名（1,1,3...），相同值并列，后续跳号
  • DENSE_RANK()：密集排名（1,1,2...），相同值并列，不跳号
  • LAG(col, n)：取当前行往前第 n 行的值（常用于环比计算）
  • LEAD(col, n)：取当前行往后第 n 行的值
  • SUM/AVG OVER：累计求和/滑动平均
典型用例：取每个用户最近一笔订单 SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY order_time DESC) AS rn
  FROM orders
) WHERE rn = 1
GROUP BY 与 HAVING 的区别
  • WHERE：在聚合前过滤原始行（不能用聚合函数）
  • GROUP BY：按指定列分组，配合聚合函数使用
  • HAVING：在聚合后过滤分组结果（可以用聚合函数）
示例：找出订单金额超过 1000 的用户 SELECT user_id, SUM(amount) AS total
FROM orders
WHERE status = 'paid' -- 先过滤（WHERE在聚合前）
GROUP BY user_id
HAVING SUM(amount) > 1000 -- 再筛选聚合结果
CTE（公共表表达式）vs 子查询
CTE 使用 WITH 语句，让复杂 SQL 更可读，支持多次引用。
CTE 写法（推荐） WITH paid_orders AS (
  SELECT * FROM orders WHERE status = 'paid'
),
user_stats AS (
  SELECT user_id, COUNT(*) AS cnt FROM paid_orders GROUP BY user_id
)
SELECT * FROM user_stats WHERE cnt > 3;
  • CTE 可以被多次引用，子查询每次都重新计算
  • CTE 语义清晰，便于调试和维护
  • 部分引擎会将 CTE 物化（缓存结果），提高性能

四、数据仓库基础 数据组织架构的核心
OLTP vs OLAP 必背
维度OLTP（在线事务）OLAP（在线分析）
典型系统MySQL / PostgreSQLHive / ClickHouse / Doris
操作类型增删改查（高频小事务）读多写少（大批量扫描）
数据量GB 级，历史数据会归档TB～PB 级，保留全量历史
查询特点按主键查单行，延迟 < 10ms全表扫描/聚合，秒～分钟级
数据模型高度范式化（3NF）反范式化（大宽表）
数仓分层架构 核心
数据源
MySQL/日志/埋点 → ODS
贴源层·原始数据 → DWD
明细层·清洗宽表 → DWS
汇总层·轻度聚合 → ADS
应用层·指标报表
  • ODS：原始数据保留，不做业务加工，保留原始字段和类型
  • DWD：清洗、标准化、关联维表，生成业务明细宽表
  • DWS：按主题汇总（用户维度/商品维度/时间维度），供多个应用共享
  • ADS：面向具体指标，直接对接 BI 工具或接口
维度建模：星型模式
由一张事实表（记录业务事件，如订单、点击）和多张维度表（描述业务主体，如用户、商品）组成。
电商星型模式举例 事实表：fact_orders（order_id, user_id, product_id, region_id, amount, dt）
维度表：dim_user（user_id, country, gender, register_date）
维度表：dim_product（product_id, category, brand, price_tier）
维度表：dim_region（region_id, country_code, timezone）
  • 事实表记录数量大（每笔订单一行），维度表量少
  • 查询时 JOIN 维度表获取描述信息
  • 为提高查询速度，常将维度字段冗余到事实表（反范式化宽表）
ETL 的含义与流程
Extract（抽取）→ Transform（转换）→ Load（加载），是数据进入仓库的必经流程。
  • E（抽取）：从 MySQL 读 Binlog、消费 Kafka 消息、解析日志文件
  • T（转换）：数据清洗（去重、过滤脏数据）、字段标准化（时区统一）、业务逻辑计算
  • L（加载）：写入 HDFS / Hive / ClickHouse，按分区存储
现代数仓也说 ELT（先加载再转换），因为大数据引擎转换能力更强，可以在仓库内直接处理

五、跨境电商核心业务指标 数据分析的业务语言
GMV
商品交易总额 (Gross Merchandise Volume)
GMV = 成交件数 × 客单价
包含取消/退款订单，是电商平台最核心的规模指标
UV
独立访客数 (Unique Visitors)
按设备ID/用户ID去重计数
同一用户访问多次只算1个UV，衡量真实用户规模
PV
页面浏览量 (Page View)
每次页面加载计1次
PV/UV = 人均浏览页数，反映用户活跃度
CVR
转化率 (Conversion Rate)
CVR = 下单用户数 / 访客数 × 100%
核心经营指标，独立站通常 1%~3% 为正常水平
AOV
平均客单价 (Average Order Value)
AOV = 总销售额 / 订单数
提升 AOV 是提升 GMV 的核心策略之一
LTV
用户生命周期价值 (Lifetime Value)
LTV = AOV × 购买频次 × 留存时长
衡量一个用户在全生命周期内创造的总价值
CAC
用户获取成本 (Customer Acquisition Cost)
CAC = 总营销费用 / 新增用户数
LTV/CAC > 3 通常认为业务健康可持续
ROAS
广告花费回报 (Return on Ad Spend)
ROAS = 广告带来的收入 / 广告花费
衡量广告投放效率，ROAS > 3 通常为盈利线
CTR
点击率 (Click-Through Rate)
CTR = 点击次数 / 展示次数 × 100%
衡量广告/商品素材的吸引力
CR
复购率 (Repurchase Rate)
CR = 有复购的用户数 / 总购买用户数
反映用户忠诚度，是评估商品/服务质量的核心
RFM
用户价值分层模型
R=最近购买时间 F=购买频次 M=累计消费
经典用户分层模型，用于精准营销和用户运营
NPS
净推荐值 (Net Promoter Score)
NPS = 推荐者% - 贬损者%
衡量用户满意度和口碑，跨境品牌极为重视

六、跨境电商业务流程与数据概念 理解业务才能设计好数据
订单生命周期 业务核心
待支付→ 用户点击购买，生成订单号
已支付→ 支付成功，GMV 计入
备货中→ 仓库收到订单，开始拣货
已发货→ 物流单号生成，追踪开始
已签收→ 用户收到货物，确认收货
售后/退款→ GMV 扣减，退款到账
注意：GMV 通常统计已支付状态，实际收入（Revenue）= GMV - 退款 - 平台费
用户行为漏斗模型 业务核心
曝光/展示
100,000 次 → CTR 2%
点击访问
2,000 UV → 加购率 15%
加入购物车
300 人 → 结账率 60%
发起结账
180 人 → 支付率 80%
完成支付
144 人 → 最终 CVR 7.2%
数据分析重点：找到漏斗中流失最大的环节（最大断层），优先优化
商品 / SKU / SPU 的区别
  • SPU（标准产品单元）：一类商品的基本信息描述，如"Nike Air Max 90 运动鞋"
  • SKU（库存最小单元）：具体规格组合，如"Nike Air Max 90 · 红色 · 42码"，库存、价格针对 SKU 管理
  • Variant（变体）：Shopify 等系统中对 SKU 的称呼
  • Listing：商品在某一平台的详情页，一个 SPU 可对应多个平台的 Listing
1个 SPU = N个 SKU（颜色 × 尺寸 = SKU数量）
库存告警、价格策略都在 SKU 粒度操作
跨境电商特有概念
  • FBA（亚马逊自营物流）：卖家发货到亚马逊仓库，由亚马逊负责配送
  • FBM（卖家自发货）：卖家自行处理物流，灵活但配送慢
  • COD（货到付款）：东南亚/中东市场常见，退货率较高
  • BFCM：Black Friday Cyber Monday，全年最大销售节点
  • 站外引流：通过 Facebook/TikTok/Google Ads 将流量引到独立站
  • 品牌出海：区别于铺货模式，建立品牌认知，提升复购率和 LTV
  • 汇率换算：多货币收款，数据分析时统一换算为基准货币（如 USD）

七、流量与广告数据基础 理解用户从哪里来
流量来源（Traffic Source）分类
  • Organic Search（自然搜索）：Google/Bing 搜索自然排名带来的流量，依靠 SEO
  • Paid Search（付费搜索）：Google Ads 关键词竞价广告（SEM）
  • Social（社交媒体）：Facebook / Instagram / TikTok 自然和付费流量
  • Email（邮件营销）：发送给已有用户的营销邮件，成本低、ROI 高
  • Direct（直接访问）：直接输入网址或书签，通常是老用户
  • Referral（引荐）：其他网站链接跳转，如联盟营销（Affiliate）
UTM 参数 — 流量追踪利器
UTM 参数附加在 URL 后面，帮助 GA/分析系统识别流量来源和广告效果。
UTM 示例 URL https://yourstore.com/products/shoes
?utm_source=facebook
&utm_medium=paid_social
&utm_campaign=bfcm_2024
&utm_content=video_ad_v2
&utm_term=running+shoes
  • utm_source：流量来源平台（facebook/google/email）
  • utm_medium：营销媒介类型（cpc/organic/newsletter）
  • utm_campaign：活动名称（便于不同活动效果对比）
广告归因模型
用户购买前可能接触多次广告，归因决定将"功劳"分配给哪次接触。
  • 最终点击归因 (Last Click)：将 100% 功劳给最后一次点击，最常用但不完整
  • 首次点击归因 (First Click)：给最初引入用户的渠道，重视品宣效果
  • 线性归因 (Linear)：均分给所有接触点，较为公平
  • 数据驱动归因：用机器学习分析每个接触点的实际贡献，最准确
跨境独立站常用 Last Click 作为主要归因，同时用 Data-driven 做参考分析
Google Analytics 核心维度
  • Session（会话）：用户在 30 分钟内的连续访问，超时则新开 Session
  • Bounce Rate（跳出率）：只访问一个页面就离开的 Session 比例，高则说明首页吸引力弱
  • Pages/Session：平均每次访问浏览页数，越高说明用户越感兴趣
  • Avg Session Duration：平均停留时长，反映内容质量
  • New vs Returning：新用户 vs 回访用户比例，体现品牌留存

八、数据分析常用思维模型 做分析比会工具更重要
拆解分析法（问题树）
当 GMV 下降时，不要直接猜原因，而是系统性拆解：
GMV 拆解树 GMV = UV × CVR × AOV

UV 下降？→ 拆分各渠道流量 → 找出哪个渠道下降
CVR 下降？→ 拆分漏斗各环节 → 找出哪一步流失大
AOV 下降？→ 看商品结构变化 → 是否低价品占比上升
A/B 测试基础概念
  • 对照组 (Control)：保持原有方案不变的用户群体
  • 实验组 (Treatment)：接受新方案（新页面/新价格/新功能）的用户群体
  • 随机分流：用用户 ID Hash 确保随机且稳定分配，同一用户始终在同一组
  • 显著性检验：p-value < 0.05 时，认为实验组与对照组差异非偶然
  • 最小样本量：需提前计算，避免过早下结论（数据太少时方差大）
同期群分析（Cohort Analysis）
将同一时间段注册/购买的用户分为一组（Cohort），追踪这批用户随时间的行为变化。
典型应用：用户留存分析 11月注册用户（Cohort）的购买留存率：
第1月购买：40%
第2月购买：20%
第3月购买：12%
第6月购买：8%
留存曲线趋于平稳说明有忠实用户群，持续下滑说明产品无法留住用户。
数据分析的常见陷阱
  • 幸存者偏差：只分析存活的用户/商品，忽略已流失的样本
  • 混淆变量：两个指标相关，但可能都是第三个因素的结果（如节假日）
  • 辛普森悖论：整体趋势与分组趋势相反，需要拆分维度分析
  • 短视 vs 长视：短期提升 CTR 可能损害长期品牌形象和复购率
  • 均值陷阱：平均客单价上涨，可能是高价商品少量拉高了均值，中位数更可靠

九、大数据工具生态速览 新人了解全貌
数据采集层
  • Flume：日志文件实时采集，写入 HDFS/Kafka
  • Canal / Debezium：MySQL Binlog 实时 CDC（变更数据捕获）
  • Kafka：分布式消息队列，数据总线，解耦生产者和消费者
  • 埋点 SDK：前端/APP 行为数据上报，如 Segment / GrowingIO
数据存储层
  • HDFS：分布式文件系统，Hive/Spark 的底层存储
  • Hive：将 SQL 转换为 MapReduce/Spark 运行，构建数仓
  • HBase：NoSQL 列存数据库，高并发随机读写，适合明细查询
  • Redis：内存 KV 存储，用于实时计数、缓存结果
  • ClickHouse/Doris：高性能 OLAP，实时分析报表
数据计算层
  • Spark：内存计算，批流一体，替代 MapReduce
  • Flink：真实时流计算，低延迟，Exactly-Once 语义
  • Presto / Trino：联邦查询，跨数据源 SQL 分析
  • dbt：数据转换工具，SQL 版本管理和血缘追踪
调度与管理层
  • Airflow：工作流调度，Python 定义 DAG，支持依赖管理
  • DolphinScheduler：国内常用的可视化任务调度系统
  • Ranger：Hadoop 权限管理，控制数据访问权限
  • Atlas：数据血缘、元数据管理，追踪数据来源
数据可视化与 BI 层
  • Superset：开源 BI 工具，支持多种数据源，美观易用
  • Metabase：更适合非技术人员的 BI，配置简单
  • Grafana：时序数据监控，常用于实时大屏
  • Tableau / Power BI：商业 BI，功能强大，跨境公司常用
  • GA4 / Shopify Analytics：电商平台自带分析工具
云平台大数据服务
  • AWS EMR：云端 Hadoop/Spark 集群，按需弹性扩缩容
  • Google BigQuery：Serverless 数仓，按查询量计费，跨境公司常用
