一、Git 版本管理基础 团队协作基石
Git 核心概念 必背
  • 仓库（Repository）：项目的根目录，包含所有文件和完整的版本历史记录
  • 分支（Branch）：独立的开发线路，可以并行工作互不干扰
  • 提交（Commit）：一次代码快照，记录修改内容、时间、作者
  • 合并（Merge）：将一个分支的修改合并到另一个分支
  • 冲突（Conflict）：两个分支修改了同一处代码，Git 无法自动合并，需手工解决
  • 暂存区（Stage）：git add 后文件进入暂存区，commit 时只提交暂存区的内容
工作区 → git add → 暂存区 → git commit → 本地仓库 → git push → 远程仓库
日常命令速查 核心
命令作用常见参数
git clone克隆远程仓库到本地git clone <url>
git add将修改加入暂存区git add . (全部) / git add <file>
git commit提交暂存区快照-m "msg"
git push推送本地提交到远程git push origin <branch>
git pull拉取远程最新代码并合并--rebase（避免多余合并提交）
git branch查看/创建/删除分支-d（删除）/ -a（查看全部）
git checkout切换分支或恢复文件-b（创建并切换）
git merge合并指定分支到当前分支--no-ff（保留合并记录）
git log查看提交历史--oneline --graph
git diff查看文件差异--staged（暂存区 vs 已提交）
git stash临时保存未提交的修改git stash pop（恢复）
分支管理策略（Git Flow）核心
main ← develop ← feature/*
main ← hotfix/*
  • main：生产分支，始终保持可发布状态，只接受 merge 不直接提交
  • develop：开发主分支，所有 feature 合并到此，测试通过后合并到 main
  • feature/*：功能分支，从 develop 拉出，完成后合并回 develop
  • hotfix/*：紧急修复分支，从 main 拉出，修复后同时合并到 main 和 develop
数仓团队常见实践 feature/dwd_order_refund → develop → 测试环境验证 → main → 调度系统上线
SQL 文件版本管理 & Code Review
ETL SQL 脚本必须纳入 Git 管理，拒绝"口口相传"和"本地文件传来传去"。
  • 目录结构：按数仓分层组织 — sql/ods/ , sql/dwd/ , sql/dws/ , sql/ads/
  • Pull Request：每次修改提交 PR，至少 1 人 Review 后才能合并
  • Review 关注点：SQL 逻辑正确性、JOIN 条件完整性、分区过滤是否遗漏、命名规范
  • CI 检查：可集成 SQLFluff 等 SQL linter 自动检查代码风格

refactor: 重构 ads_gmv_daily CTE 结构

二、任务调度系统 ETL 自动化运行
为什么需要任务调度？ 必背
数据仓库中的 ETL 任务之间存在严格的依赖关系，不能简单用 crontab 管理。
  • 依赖关系：DWD 层必须等 ODS 层完成后才能运行
  • 失败重试：某个任务失败时自动重试，通知相关人
  • 并行调度：无依赖的任务可以同时运行，提升效率
  • 回溯补数：某天数据异常，需要从某个节点开始重跑下游全部任务
  • 可视化监控：通过 UI 查看任务状态、运行时间、历史记录
crontab 的问题：无法表达依赖、无失败重试、无可视化、无法补数
DAG（有向无环图）概念
调度系统的核心数据结构 — 每个节点代表一个任务，箭头代表依赖方向，不能有环。
ODS 订单 → DWD 订单宽表 → DWS 用户汇总 → ADS GMV 报表
ODS 用户 → DWD 订单宽表
DWD 订单宽表同时依赖 ODS 订单和 ODS 用户 → 两个上游都完成后才触发
Airflow 入门 核心
  • DAG 定义：用 Python 代码定义任务和依赖关系，版本可控
  • Operator 类型：BashOperator（Shell命令）、PythonOperator（Python函数）、HiveOperator（Hive SQL）、SparkSubmitOperator
  • 依赖设置：task_a >> task_b 表示 a 完成后再执行 b
  • 调度周期：cron 表达式，如 "0 2 * * *" 表示每天凌晨 2 点
  • 回填（Backfill）：可以指定日期范围重跑历史任务
Airflow DAG 示例 with DAG("dw_daily", schedule="0 2 * * *") as dag:
  ods = BashOperator(task_id="ods_sync", ...)
  dwd = HiveOperator(task_id="dwd_order", ...)
  dws = HiveOperator(task_id="dws_user", ...)
  ods >> dwd >> dws

示例表名拆解说明

分区规范与 SQL 代码规范
  • 分区字段：通常用 dt（日期分区），格式 yyyy-MM-dd 或 yyyyMMdd
  • 二级分区：高频数据可按 小时（hour）或 地区（region）二次分区
  • 分区裁剪：查询时 WHERE 必须带分区条件，避免全表扫描
SQL 代码规范 -- 1. 关键字大写：SELECT / FROM / WHERE / JOIN / GROUP BY
-- 2. JOIN 条件独立行：
  SELECT a.user_id, b.user_name
  FROM dwd_order AS a
  LEFT JOIN dim_user AS b
    ON a.user_id = b.user_id
-- 3. 复杂逻辑用 CTE 拆分，避免嵌套超过 3 层
-- 4. 每段逻辑加注释说明业务含义
指标口径统一管理
指标口径不统一是数仓最常见的问题 — 同一个"GMV"在不同报表里数字不同。
  • 指标字典：统一维护每个指标的名称、定义、计算公式、数据来源
  • 口径文档化：GMV 是否包含退款？是否含税？是否含运费？
  • DWS 层收口：核心指标只在 DWS 层计算一次，ADS 层直接引用，禁止重新计算
  • 指标 Owner：每个核心指标指定一个负责人，修改需走审批流程
指标定义示例 指标名：GMV（商品交易总额）
定义：已支付订单的商品金额合计（含已退款，不含运费）
公式：SUM(pay_amount) WHERE order_status IN ('paid','shipped','delivered','refunded')
数据源：dws_trade_order_summary_d.gmv_amount

四、开发流程与协作 标准化保障质量
数仓开发标准流程 必背
需求分析 → 技术方案 → 编码开发 → 自测验证 → Code Review → 上线发布 → 监控告警
  • 需求分析：明确业务方需要什么指标？口径如何定义？时效要求？
  • 技术方案：设计表结构、数据来源、ETL 逻辑、调度依赖，输出方案文档
  • 编码开发：在 feature 分支上编写 SQL，遵循命名和代码规范
  • 自测验证：开发环境跑通 SQL，验证数据准确性
  • Code Review：提交 PR，由同事审查逻辑、规范、性能
  • 上线发布：合并到 main，配置调度任务，首次手动触发验证
  • 监控告警：配置数据质量监控规则和失败告警
测试方法 核心
测试类型方法SQL 示例
行数验证对比源表与目标表行数SELECT COUNT(*) 对比上下游
空值检查关键字段不应为 NULLWHERE user_id IS NULL

上线 Checklist & 变更管理
  • □ SQL 已通过 Code Review
  • □ 开发环境自测数据正确
  • □ 分区条件已添加，无全表扫描风险
  • □ 调度依赖已配置，上游任务完成后触发
  • □ 告警已配置，失败/超时通知到人
  • □ 数据质量监控规则已添加
  • □ 数据字典已更新

灰度上线 → 先跑一天验证，确认无误后切换正式调度

五、数据文档与数据字典 知识沉淀不可少
为什么要写文档？ 必背
  • 降低沟通成本：新人入职可以直接查文档了解表结构和业务口径
  • 避免知识孤岛：不写文档 = 只有 SQL 作者知道逻辑，他离职了就是黑盒
  • 减少重复劳动：业务方反复问"这个指标怎么算？" → 给他文档链接
  • 提升信任度：清晰的口径文档让业务方相信数据是可靠的
  • 支持影响分析：修改表结构前，通过文档和血缘确认影响范围

七、常见开发陷阱与避坑指南 前人踩坑后人避
DROP TABLE 误操作 高危
生产环境误执行 DROP TABLE 是数据团队最严重的事故之一。
  • 防御措施：生产账号只授 SELECT / INSERT 权限，禁止 DROP / TRUNCATE
  • 操作规范：执行高风险 SQL 前必须先在测试环境验证
  • 回收站：Hive 开启回收站（Trash），删除的表保留 7 天可恢复
  • 备份机制：核心表每日快照备份，保留至少 30 天
真实案例 开发同学在生产终端执行了 DROP TABLE dwd_trade_order（本想在测试环境执行）
→ 所有下游报表中断 → 紧急从 HDFS 回收站恢复 → 耗时 4 小时
教训：养成检查当前数据库连接的习惯，高危操作前先 SELECT 确认
动态分区小文件爆炸 高危
Hive / Spark 动态分区写入时，如果分区键基数很高，会产生大量小文件。
  • 原因：每个 Task 对每个分区值生成一个文件 → Task数 × 分区数 = 文件总数
  • 后果：HDFS NameNode 内存被撑爆，后续查询性能极差
  • 解决方案：写入前 DISTRIBUTE BY 分区键合并；写入后执行小文件合并任务
200 个 Task × 365 天分区 = 73,000 个小文件 → NameNode 承压严重
Spark：设置 spark.sql.shuffle.partitions 和 coalesce 控制输出文件数
时区问题（UTC vs 北京时间）
跨境电商涉及多个时区，时区处理不当会导致数据错位。
  • 常见问题：数据源是 UTC 时间，报表按北京时间（UTC+8）统计，分区归属错误
  • 错误表现：12月31日 20:00 UTC 的订单被归到 12 月 31 日（应为 1 月 1 日北京时间）
  • 最佳实践：ODS 层保留原始时区，DWD 层统一转换为业务时区并存储转换后的日期分区
  • 转换函数：from_utc_timestamp(ts, "Asia/Shanghai")
团队约定：所有 _time 字段存储 UTC，所有 _date / dt 字段存储北京时间日期
NULL 参与计算的陷阱
SQL 中 NULL 不等于空字符串也不等于 0，任何值与 NULL 运算结果都是 NULL。
  • NULL + 100 = NULL：金额字段为 NULL 时，SUM 不受影响，但算术运算会传染
  • NULL != NULL：WHERE col != 'abc' 不会返回 col 为 NULL 的行
  • COUNT(col) vs COUNT(*)：COUNT(col) 会跳过 NULL，COUNT(*) 统计所有行
  • JOIN 陷阱：NULL = NULL 返回 FALSE，含 NULL 的 JOIN Key 不会匹配
防御方式 COALESCE(amount, 0) — 将 NULL 转为 0
NVL(col, 'unknown') — Hive 中将 NULL 替换为默认值
WHERE col IS NULL — 判断 NULL 用 IS 而不是 =
SORT BY vs ORDER BY
语句作用范围性能使用场景
ORDER BY全局排序（单个 Reducer）数据量大时极慢最终输出需要全局有序
SORT BY每个 Reducer 内部排序可并行，性能好中间过程局部排序即可
DISTRIBUTE BY按指定 Key 分发到 Reducer控制数据分布配合 SORT BY 使用
CLUSTER BY= DISTRIBUTE BY + SORT BY简写分发和排序字段相同时
陷阱：对 10 亿行数据用 ORDER BY → 只有 1 个 Reducer → 任务卡死 OOM
建议：大数据量排序用 SORT BY + DISTRIBUTE BY，或 ROW_NUMBER() 窗口函数
忘记分区条件全表扫描 高危
查询 Hive/Spark 表时忘记加分区条件，导致扫描全量历史数据。
  • 后果：一个简单查询扫描 TB 级数据，占用大量集群资源，影响其他任务
  • 防御 — 严格模式：Hive 开启 hive.strict.checks.no.partition.filter=true
  • 防御 — CR 审查：Code Review 时检查每个 FROM 后面是否有分区条件
  • 防御 — 成本估算：执行前用 EXPLAIN 查看扫描数据量
错误示例 SELECT COUNT(*) FROM dwd_trade_order_detail_d;
-- 扫描了 3 年 × 365 天所有分区 = 1.2TB

正确示例 SELECT COUNT(*) FROM dwd_trade_order_detail_d WHERE dt = '2025-03-10';
