Q1: 请说明 useState、useReducer、useRef 的使用场景区别，以及它们触发 re-render 的差异。
基础 / 初级

考察关键点

  • useState：简单独立值，setter 调用 → 组件 re-render。
  • useReducer：多个相互关联的状态，通过 action 集中管理，逻辑更清晰，便于测试；reducer 可抽离。
  • useRef：存可变值但不触发 re-render；持有 DOM 引用；保存动画实例、定时器 ID。
  • 何时从 useState 升级到 useReducer：当 setter 调用超过 3 个、状态之间有依赖关系时。

4–6 分
知道三者基本用法，useRef 不触发渲染，但场景区分不够清晰。

Q2: useEffect、useLayoutEffect、useMemo、useCallback 的执行时机和性能陷阱，分别是什么？
中级

考察关键点

  • useEffect：Paint 之后异步执行，适合副作用（接口请求、订阅）；不适合 DOM 读写（会闪）。
  • useLayoutEffect：DOM 变更后、Paint 之前同步执行；适合测量 DOM、动画初始化；SSR 时会报警告（用 useIsomorphicLayoutEffect 处理）。
  • useMemo：缓存计算结果；滥用会增加内存开销（每次 re-render 检查依赖本身有成本）；仅用于昂贵计算。
  • useCallback：缓存函数引用；子组件用 React.memo 包裹时才有意义；否则只是额外开销。

评分标准 (9分)

8–9 分
精确描述 useLayoutEffect 执行时序；知道 useMemo/useCallback 存在负面成本，不盲目使用；提到 SSR 兼容处理。
4–7 分
区分 useEffect 和 useLayoutEffect，知道 useMemo 缓存，但对"滥用有害"的理解不深。
0–3 分
到处用 useMemo/useCallback "优化"，认为加了肯定更快。
查看参考答案与执行时序图

执行顺序（一帧内）：

  1. React 渲染（虚拟 DOM diff）
  2. DOM 更新（真实 DOM 变化）
  3. useLayoutEffect 同步执行（← 读取 DOM 尺寸、初始化动画）
  4. 浏览器 Paint（像素绘制到屏幕）
  5. useEffect 异步执行（← 发接口、监听 WebSocket）

useMemo/useCallback 使用原则：先正常写，性能有问题再加；在 DevTools Profiler 中确认组件确实过度渲染后再包裹 memo。

Q3: 深入理解 React 的 Fiber、调度机制 (Concurrent Mode) 和 Batching (批量更新) — 对性能有何影响？
高级 / 专家

考察关键点

  • Fiber 架构：每个组件对应一个 Fiber 节点（链表），将渲染拆成可中断的小任务（Work Units），让浏览器在空闲时执行。
  • 时间切片 (Time Slicing)：React 将渲染任务切成 5ms 的时间片，用 MessageChannel 调度，不阻塞用户交互。
  • React 18 自动批处理：React 17 仅在事件处理中批量更新，18 中任何地方（setTimeout、Promise、原生事件）都自动批量更新。
  • startTransition：标记低优先级更新（如搜索过滤），不阻塞高优先级的输入响应。

7–8 分

4–6 分
知道 Concurrent Mode 和批量更新概念，但实战中不清楚何时使用 startTransition。
0–3 分
认为 React 渲染是同步阻塞的，不了解调度机制。

Q5: 如何避免 React 组件的过度渲染（Over-rendering）？React.memo、useMemo、useCallback 的正确配合方式。
性能优化

考察关键点

  • Re-render 触发条件：父组件渲染 → 子组件默认全部渲染（不管 props 有没有变化）。
  • React.memo：浅比较 props，props 无变化则跳过渲染；与 useCallback 配合才有效（函数引用每次都是新的）。
  • 三件套配合：父传给 memo 子组件的函数用 useCallback、对象用 useMemo 包裹，确保引用稳定。
  • 何时不需要优化：叶子组件渲染极快，优化反而增加内存和 GC 压力。
  • 诊断工具：React DevTools Profiler → 找"为什么渲染了"。

评分标准 (10分)

8–10 分
能解释浅比较的局限性；知道何时该优化、何时优化反而有害；使用 Profiler 定位问题而非凭感觉优化。
4–7 分
知道 memo + useCallback 配合，但不清楚优化的成本和边界。
0–3 分
对所有组件加 memo，所有函数加 useCallback，认为没有副作用。
查看参考代码与优化决策树 
// 父组件
const Parent = () => {
  const [count, setCount] = useState(0);

  // ✅ 用 useCallback 稳定函数引用（配合 ChildMemo 使用）
  const handleAdd = useCallback(() => setCount(c => c + 1), []);

  return <>

    <ChildMemo onAdd={handleAdd} config={config} />
    <SimpleChild /> {/* 渲染极快，不需要 memo */}
  </>;
};

// 子组件用 React.memo 包裹才能利用稳定引用
const ChildMemo = React.memo(({ onAdd, config }) => {
  console.log('ChildMemo rendered'); // 只在 onAdd/config 变化时触发
  return <div onClick={onAdd}>{config.theme}</div>;
});

Q6: 产品详情页：用户切换 SKU 选项后，价格、图片、参数表、库存状态、样式主题同时联动更新，如何设计状态结构？
核心实战

考察关键点

  • 单一数据源：所有联动数据以选中 SKU ID 为核心，其他数据通过 selector 派生，不在 state 中冗余存储。

  • 避免状态爆炸：不要为每个联动项（价格、库存、颜色）各自维护一个 state，否则多次 setState 引发多次渲染。
  • 数据预处理：接口返回数据在进入 store 前做一次规范化（Normalize），建立 SKU Map，切换时 O(1) 查询。

Zustand Store 设计（产品详情页）：

// store/productStore.ts
interface ProductState {
  product: Product;              // 原始产品数据（含所有 SKU）
  selectedOptions: OptionMap;    // { color: 'red', size: 'M' }
  skuMap: Record<string, Sku>;   // 预构建的 SKU 快查表
}

export const useProductStore = create<ProductState>()((set, get) => ({
  product: null,
  selectedOptions: {},
  skuMap: {},

  setOption: (key, value) => {
    set(state => ({
      selectedOptions: { ...state.selectedOptions, [key]: value }
    }));
  },
}));

// 派生 selector（组件中按需订阅，不触发无关组件更新）
export const useCurrentSku = () =>
  useProductStore(state => {
    const key = buildSkuKey(state.selectedOptions); // 'red-M'
    return state.skuMap[key];
  });

// 各组件只订阅自己关心的数据
const PriceDisplay = () => {
  const sku = useCurrentSku();       // ← 只在 sku 变化时渲染
  return <div>¥{sku?.price}</div>;
};

关键优化：接口数据进入 store 前构建 skuMap，切换选项时 O(1) 查找，不需要每次遍历数组。

Q9: 详细对比 useContext、Redux Toolkit、Zustand 的原理差异 — 更新 state 时，分别会导致哪些组件 re-render？如何精确控制？
状态管理核心

考察关键点

  • 选型建议：独立站/中小项目首选 Zustand；大型企业项目需要严格数据流审计选 Redux；局部共享用 Context。

Q10: Zustand 进阶：如何做 store 分片（Slices）、如何配合 immer 处理深层嵌套更新、如何实现 persist 持久化？
Zustand 深度

考察关键点

  • Slices 模式：将大 store 拆成多个 slice 函数，合并到一个 create 调用中，防止 store 臃肿。
  • immer 中间件：处理深层嵌套对象更新时不需要手写展开语法，直接 mutate 草稿对象（不可变更新的语法糖）。

// hooks/useScrollAnimation.ts
const useScrollAnimation = (config: AnimConfig) => {
  const ref = useRef(null);
  const ctxRef = useRef(null);   // GSAP Context，用于批量清理

  // SSR 安全 + DOM 保证存在
  useIsomorphicLayoutEffect(() => {
    if (!ref.current) return;

    // 检查无障碍偏好
    const reduced = window.matchMedia(
      '(prefers-reduced-motion: reduce)'
    ).matches;
    if (reduced) return;

    // GSAP Context 作用域 — 自动收集所有动画便于一次性清理
    ctxRef.current = gsap.context(() => {
      gsap.from(ref.current, {
        ...config.from,
        scrollTrigger: {
          trigger: ref.current,
          start: config.start ?? 'top 80%',
          toggleActions: 'play none none reverse',
        },
      });
    });

    return () => ctxRef.current?.revert(); // 彻底清理，防止内存泄漏
  }, []);  // 空依赖：只初始化一次

  return ref;
};

};

const useServerCountdown = (endTime: string) => {
  const [timeLeft, setTimeLeft] = useState(0);
  const offsetRef = useRef(0);

  useEffect(() => {
    const t0 = Date.now();
    fetch('/api/server-time').then(async res => {
      const t1 = Date.now();
      const serverNow = Number(res.headers.get('X-Server-Time'));
      // 补偿网络往返延迟的一半，使偏差更精准
      offsetRef.current = serverNow - t0 - (t1 - t0) / 2;

      const tick = () => {
        const correctedNow = Date.now() + offsetRef.current;
        const remaining = new Date(endTime).getTime() - correctedNow;
        setTimeLeft(Math.max(0, remaining));
      };
      tick();
      const id = setInterval(tick, 1000);
      return () => clearInterval(id);
    });
  }, [endTime]);

  return timeLeft; // 毫秒数，组件自行格式化为 HH:MM:SS

Q19: 深度考察 IntersectionObserver — 如何封装通用 useInView Hook，同时满足懒加载、动画触发、曝光埋点三种场景？参数如何精确控制？
可见性检测 · 懒加载 · 埋点

考察关键点

  • 三个关键参数：root（观察相对的祖先容器，默认视口）、rootMargin（提前/延后触发边界，如 "200px 0px" 提前 200px 开始加载）、threshold（触发时目标元素的可见比例，0~1，数组则多次触发）。
  • 三种场景的参数差异：懒加载 → threshold=0 + rootMargin="300px"（提前加载）；动画触发 → threshold=0.2（至少 20% 可见时才播）；曝光埋点 → threshold=0.5 + 只触发一次（用户真正"看到了"）。
  • once 选项：动画和埋点通常只触发一次（进入视口后 disconnect）；懒加载同理；只有无限滚动等需要反复触发（不 disconnect）。
  • SSR 兼容：服务端没有 IntersectionObserver，Hook 内必须判断 typeof IntersectionObserver !== 'undefined'，否则 Next.js 服务端渲染时报错。
  • 曝光埋点的精确性：不能在组件 mount 时就上报（元素可能在折叠以下用户没看到）；必须通过 IntersectionObserver 确认真正进入视口 + 停留超过 500ms 才算有效曝光（防止快速滚动误触）。

1. 通用 useInView Hook（满足所有场景）：

// hooks/useInView.ts
interface UseInViewOptions {
  root?: Element | null;
  rootMargin?: string;    // 提前/延后触发：正值提前，负值延后
  threshold?: number | number[];
  once?: boolean;         // true = 触发一次后自动 disconnect
  delay?: number;         // 需停留 N ms 后才确认"已看到"
}

const useInView = (options: UseInViewOptions = {}) => {
  const {
    rootMargin = '0px',
    threshold = 0,
    once = true,
    delay = 0,
  } = options;

  const ref = useRef<Element>(null);
  const [inView, setInView] = useState(false);
  const timerRef = useRef<ReturnType<typeof setTimeout>>(null);

  useEffect(() => {
    // SSR 兼容
    if (typeof IntersectionObserver === 'undefined') return;
    const el = ref.current;
    if (!el) return;

    const observer = new IntersectionObserver(([entry]) => {
      if (entry.isIntersecting) {
        if (delay > 0) {
          // 停留 delay ms 后才确认（防止快速滚过误触）
          timerRef.current = setTimeout(() => {
            setInView(true);
            if (once) observer.disconnect();
          }, delay);
        } else {
          setInView(true);
          if (once) observer.disconnect();
        }
      } else {
        // 离开视口：清除未完成的 delay 计时
        if (timerRef.current) clearTimeout(timerRef.current);
      }
    }, { rootMargin, threshold });

    observer.observe(el);
    return () => {
      observer.disconnect();
      if (timerRef.current) clearTimeout(timerRef.current);
    };
  }, [rootMargin, threshold, once, delay]);

  return { ref, inView };
};

    name: metric.name,        // 'LCP' | 'CLS' | 'INP' ...
    value: metric.value,      // 实测值
    rating: metric.rating,    // 'good' | 'needs-improvement' | 'poor'
    delta: metric.delta,      // 相比上次的变化量
    id: metric.id,
    // 附带设备/地区信息，用于分组分析
    ua: navigator.userAgent,
    locale: navigator.language,
    url: location.pathname,
  });

  navigator.sendBeacon('/api/analytics/vitals', body);
};

// 在 _app.tsx 或 layout.tsx 中注册
export const initWebVitals = () => {
  onCLS(reportMetric);   // 布局稳定性（动画 height 变化是 CLS 主要来源）
  onLCP(reportMetric);   // 最大内容渲染（首屏英雄图加载速度）
  onINP(reportMetric);   // 交互响应（JS 动画阻塞主线程会影响此指标）
  onFCP(reportMetric);   // 首次内容渲染
  onTTFB(reportMetric);  // 首字节时间（服务端响应速度）
};

3. 上报分析策略建议：

  • P75 分位数比均值更真实，过滤极端值影响（极少数超慢用户拉高均值）
  • 分组维度：按地区（欧美 vs 亚洲）、设备类型（iOS / Android / PC）、页面类型分别分析，定位特定用户群的性能问题
  • 采样率：高流量时只上报 10–30% 用户的数据，节省带宽和存储成本
  • 告警阈值：LCP > 4s 或 INP > 500ms 时触发 Slack/邮件告警，第一时间感知线上回归
