Q1: 为什么 transform 和 opacity 动画性能最高？请详述浏览器渲染流水线 (Rendering Pipeline)。
核心必问

考察关键点

  • 流水线阶段：Parse HTML → Style → Layout (Reflow) → Paint (Repaint) → Composite。
  • 触发层级：改变 width/left 触发 Layout；改变 color 触发 Paint；transform/opacity 仅触发 Composite。
  • 合成线程：Composite 在独立线程运行，不阻塞主线程，即使 JS 阻塞也能保持 60fps。
  • GPU 纹理上传：合成层变成 GPU 纹理，变换操作直接由 GPU 矩阵运算完成。

评分标准 (10分)

8–10 分
能完整描述像素管道全流程；区分主线程/合成线程；解释 GPU 纹理混合过程。
4–7 分
知道 GPU 加速、Layout/Paint 区别，但对线程模型描述模糊。
0–3 分
只知道"transform 快"，不能解释原因。
查看参考答案解析

浏览器完整像素管道：

  1. Style：计算每个元素最终 CSS 值。
  2. Layout：计算元素尺寸与位置（最昂贵）。
  3. Paint：将元素绘制到位图（像素填充）。
  4. Composite：将多个合成层合并输出到屏幕（纯 GPU 操作）。

关键：transform/opacity 的变化完全绕过 Layout 和 Paint，仅由 GPU 矩阵乘法完成，且运行在独立的合成线程，即使主线程被 JS 长任务阻塞，动画依然流畅。

Q2: will-change 属性的原理是什么？滥用会带来什么问题？
性能优化

考察关键点

  • 提前提升合成层：浏览器提前为元素创建独立纹理（GPU 显存），避免动画时临时创建的开销。
  • VRAM 占用：每个合成层消耗显存，大量 will-change 会导致移动端 OOM（内存不足）Crash。
  • 层爆炸 (Layer Explosion)：子元素可能被意外提升到独立层，触发大量 Composite。
  • 正确用法：动画前通过 JS 动态添加，动画结束后移除；不使用 * { will-change: transform }。

评分标准 (10分)

8–10 分
明确指出 VRAM 爆炸风险；知道 translateZ(0) 是旧版 Hack；能区分 will-change 与 transform: translate3d 的异同。
4–7 分
知道它提升 GPU 加速，但不清楚副作用与使用时机。
0–3 分
不了解或认为"加上只有好处"。
查看参考答案解析

最佳实践：

// 动画开始时添加
el.style.willChange = 'transform';
el.addEventListener('transitionend', () => {
  el.style.willChange = 'auto'; // 动画结束后还原
}, { once: true });

警惕场景：在 CSS 文件中对大量元素静态设置 will-change，会让整个页面所有元素都占用 GPU 显存，移动端极易崩溃。

Q3: steps() 和 cubic-bezier() 的数学含义是什么？为什么 ease-out 感觉"自然"？
数学原理

考察关键点

  • 贝塞尔本质：时间轴 t(0→1) 映射到属性值 p(0→1) 的曲线，两个控制点决定斜率（速度）变化。
  • ease-out 感觉自然：符合物理摩擦力：先快后慢，像滑动停止，符合人眼"惯性期待"。
  • steps() 阶梯函数：将时间切割成 N 段，用于 Sprite Sheet 逐帧动画；steps(12, end)。
  • spring 与 bezier 的根本区别：bezier 时长固定，spring 基于物理方程，时长由速度动态决定。

8–10 分
能解释斜率代表速度；用 steps 实现逐帧动画；对比 spring 物理与贝塞尔的差异（speed → duration vs duration fixed）。
4–7 分
知道 ease-out "先快后慢"，能用 cubic-bezier 调节效果。
0–3 分
只会用预设关键词（ease/linear），不理解背后数学。
查看参考答案解析

cubic-bezier(x1, y1, x2, y2) 快速记忆：

  • ease = cubic-bezier(0.25, 0.1, 0.25, 1) — 慢快慢
  • ease-in = cubic-bezier(0.42, 0, 1, 1) — 先慢（加速入场）
  • ease-out = cubic-bezier(0, 0, 0.58, 1) — 先快后慢（最像物理）
  • 弹簧效果：cubic-bezier(0.34, 1.56, 0.64, 1) — Y 值超过 1，模拟过冲

steps 应用：雪碧图逐帧动画，精灵动画走路循环。

Q4: 算法题：实现帧率无关的 Lerp (Damp) 函数。简单 Lerp 在 120Hz 屏幕上为什么会"跑得更快"？
基础算法

考察关键点

  • 基础公式：lerp(a, b, t) = a + (b - a) * t。
  • 帧率耦合问题：每帧执行 cur += (target - cur) * 0.1，60hz 每秒执行 60 次，120hz 执行 120 次，速度翻倍。
  • 修复方案：引入 deltaTime，用指数衰减公式：t = 1 - Math.exp(-lambda * dt)。
  • 应用场景：鼠标跟随阻尼、相机平滑、平滑滚动位置。

评分标准 (10分)

8–10 分
能手写帧率无关的 damp 函数；解释 Math.exp 指数衰减原理；知道 lambda 参数控制"阻尼强度"。
4–7 分
知道 Lerp 公式和用途，但未考虑帧率差异。
0–3 分
不了解 Lerp，或认为用 CSS transition 可替代一切。
查看参考代码 
// 基础版（帧率相关 — 不推荐）
const lerp = (a, b, t) => a + (b - a) * t;

// 进阶版：帧率无关 Damp（推荐）
// lambda: 阻尼系数（越大越快追上目标，通常 1–20）
// dt: 上一帧到当前帧的时间差（单位：秒）
const damp = (a, b, lambda, dt) =>
  lerp(a, b, 1 - Math.exp(-lambda * dt));

// 使用示例 (requestAnimationFrame loop)
let prev = performance.now();
const tick = () => {
  const now = performance.now();
  const dt = (now - prev) / 1000; // 转换为秒
  prev = now;

  curX = damp(curX, targetX, 8, dt);
  el.style.transform = `translateX(${curX}px)`;
  requestAnimationFrame(tick);
};
Q5: 场景题：如何检测鼠标是否在一个经过 CSS rotate 旋转后的矩形元素内？(OBB 碰撞检测)
图形学难点

考察关键点

  • AABB vs OBB：getBoundingClientRect 返回的是 AABB（轴对齐包围盒），旋转后精度差。
  • 逆变换法：将鼠标坐标通过矩形变换矩阵的逆矩阵还原到元素局部坐标系，再做 AABB 检测。
  • 矩阵获取：getComputedStyle(el).transform 拿到 matrix 值，解析旋转角度。
  • 分离轴定理 (SAT)：另一种通用方案，但对简单矩形来说逆变换更简洁。

评分标准 (10分)

8–10 分
给出完整的逆变换法方案；知道 DOMMatrix API；提到 SAT 作为通用备选；指出 getBoundingClientRect 的精度缺陷。
4–7 分
知道 getBoundingClientRect 不够用，有坐标转换的思路但细节模糊。
0–3 分
直接用 getBoundingClientRect，忽略旋转影响。
查看参考答案解析

逆变换法核心代码：

function isPointInRotatedRect(el, mouseX, mouseY) {
  const rect = el.getBoundingClientRect();
  const cx = rect.left + rect.width / 2;
  const cy = rect.top + rect.height / 2;

  // 获取元素旋转角度
  const style = getComputedStyle(el);
  const matrix = new DOMMatrix(style.transform);
  const angle = Math.atan2(matrix.m12, matrix.m11); // 弧度

  // 将鼠标坐标平移到以元素中心为原点
  const dx = mouseX - cx;
  const dy = mouseY - cy;

  // 逆旋转（乘以负角度旋转矩阵）
  const cos = Math.cos(-angle);
  const sin = Math.sin(-angle);
  const localX = dx * cos - dy * sin;
  const localY = dx * sin + dy * cos;

  // 在局部坐标系做 AABB 检测
  return (
    Math.abs(localX) <= rect.width / 2 &&
    Math.abs(localY) <= rect.height / 2
  );
}

粒子系统

考察关键点

  • Canvas vs DOM：超过 100 个粒子必须用 Canvas；超过 5000 个考虑 WebGL Instancing。

// 三次贝塞尔公式
const bezier = (p0, p1, p2, p3, t) => {
  const mt = 1 - t;
  return mt**3 * p0 + 3 * mt**2 * t * p1 +
         3 * mt * t**2 * p2 + t**3 * p3;
};

// 对象池：预创建 N 个粒子
class ParticlePool {
  constructor(size) {
    this.pool = Array.from({ length: size }, () => ({
      x: 0, y: 0, t: 0, speed: 0, alive: false
    }));
  }
  get() {
    return this.pool.find(p => !p.alive) || this.pool[0];
  }
  release(p) { p.alive = false; }
}

等弧长采样：预先在路径上采样 1000 个点存入 LUT，用线性插值按距离均匀取点，避免"曲率大的地方粒子慢"的问题。

Q7: 什么是四元数 (Quaternion)？在 3D 动画中为什么比欧拉角 (Euler Angles) 更好？

Q8: 在 Next.js 中实现高性能 Scroll-driven Animation，对比 GSAP / Framer Motion / 原生 CSS 的选型。
技术选型

考察关键点

  • CSS Scroll Timeline（原生）：零 JS 开销，在合成线程运行，未来趋势，但兼容性还需 Polyfill。

  • React 核心陷阱：绝对不能在 onScroll 回调中 setState；应使用 useRef 或 MotionValue 直接操作 DOM。

8–10 分
能给出不同场景下的最优选择；知道 CSS Scroll Timeline 的线程优势；推荐 Lenis 做底层平滑滚动；理解 Direct Manipulation 绕过 React Render Cycle。

0–3 分

查看参考答案解析

选型决策树：

  • 追求极致性能 + 未来兼容 → CSS Scroll Timeline
  • 复杂叙事型官网 + Pin 效果 → GSAP ScrollTrigger
  • React 组件内联动画 → Framer Motion useScroll

Q11: FLIP 技术（First Last Invert Play）原理与实战 — 手写伪代码并指出 Scale 变换的子元素扭曲问题。
性能模式

考察关键点

  • 解决的问题：DOM 布局改变（如 Flex 排序、Grid 重排）无法直接 CSS transition，因为新位置在下一帧才生效。
  • 四步流程：① 记录初始位置 (First) → ② 触发布局变化 → ③ 记录新位置 (Last) → ④ 用 transform 将元素"拉回"初始位置 (Invert) → ⑤ 移除 transform 触发过渡 (Play)。
  • Scale 扭曲问题：父元素做 scale 变换时，子元素会跟着被拉伸，需要给子元素应用反向 Scale（Inverse Scale）修正。
  • Framer Motion：layout prop 自动处理 FLIP + Scale 修正。

评分标准 (10分)

8–10 分
能完整手写 FLIP 四步伪代码；指出 Scale 修正问题（反向 scaleX/Y = 1 / parentScaleX）；提到 Framer Motion layout 的底层就是 FLIP。
4–7 分
理解 FLIP 原理，知道用 getBoundingClientRect，但细节不完整。
0–3 分
不了解 FLIP，用 CSS transition: all 0.3s 解决布局动画。
查看参考代码 
async function flipAnimate(el, changeLayout) {
  // 1. First — 记录初始位置
  const first = el.getBoundingClientRect();

  // 2. 触发布局变化
  changeLayout();

  // 3. Last — 记录新位置（需等 DOM 更新）
  await Promise.resolve(); // 或 requestAnimationFrame
  const last = el.getBoundingClientRect();

  // 4. Invert — 用 transform 还原到初始位置
  const dx = first.left - last.left;
  const dy = first.top - last.top;
  const dw = first.width / last.width;
  const dh = first.height / last.height;

  el.style.transform =
    `translate(${dx}px, ${dy}px) scale(${dw}, ${dh})`;
  el.style.transition = 'none';

  // 5. Play — 移除 transform，触发过渡动画
  requestAnimationFrame(() => {
    el.style.transition = 'transform 0.4s ease-out';
    el.style.transform = 'none';
  });
}

查看参考代码（GLSL Fragment Shader） 
// Fragment Shader — 水波纹效果
uniform sampler2D uTexture;
uniform float uTime;
varying vec2 vUv;  // 从 vertex shader 传入的 UV 坐标

void main() {
  vec2 uv = vUv;

  // sin 波纹扰动 UV 坐标
  float dist = length(uv - 0.5);
  float wave = sin(dist * 30.0 - uTime * 3.0) * 0.02;
  uv += wave;

  // 用扰动后的 UV 采样纹理
  gl_FragColor = texture2D(uTexture, uv);
}

Perlin Noise 升级版：用噪声函数替代 sin 可获得更有机的水波效果，避免规则重复感。

考察关键点

  • Canvas 序列帧：用 Canvas drawImage 绘制图片序列，Sticky 容器固定，ScrollY 映射帧索引。
  • 预加载策略：Promise.all 并发加载，createImageBitmap 在解码线程处理，不阻塞主线程。
  • 移动端降级：检测 navigator.connection.effectiveType 或 UA，2G/3G 降为低分辨率序列或静态图/视频。
  • 内存管理：视口外的帧 ImageBitmap.close() 释放；分段加载（不一次性加载全部 150 帧）。
  • Retina 处理：canvas.width = clientWidth * devicePixelRatio，确保高清屏不模糊。

// 1. Sticky 容器结构
// <div style="height: 500vh">
//   <canvas style="position: sticky; top: 0" />
// </div>

// 2. 分段预加载 + createImageBitmap
const loadFrames = async (start, end) => {
  const promises = Array.from({ length: end - start }, (_, i) => {
    return fetch(`/frames/${start + i}.webp`)
      .then(r => r.blob())
      .then(blob => createImageBitmap(blob)); // 解码线程
  });
  return Promise.all(promises);
};

// 3. 滚动映射帧索引
const onScroll = () => {
  const section = container.getBoundingClientRect();
  const progress = -section.top / (section.height - window.innerHeight);
  const frameIdx = Math.floor(
    Math.max(0, Math.min(1, progress)) * (TOTAL_FRAMES - 1)
  );
  ctx.drawImage(frames[frameIdx], 0, 0, canvas.width, canvas.height);
};

// 4. Retina 适配
const dpr = window.devicePixelRatio || 1;
canvas.width = canvas.clientWidth * dpr;
canvas.height = canvas.clientHeight * dpr;
ctx.scale(dpr, dpr);

移动端降级策略：通过 navigator.connection.effectiveType 判断网速，慢速网络直接 autoplay 视频替代序列帧。
